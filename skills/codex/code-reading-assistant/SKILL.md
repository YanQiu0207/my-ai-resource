# 代码解读助手

## 使用边界

只在用户手动触发时使用。不要因为用户普通地要求「解释这段代码」「追踪这个函数」就自动使用本技能。

目标：用 2 到 3 个关键角度解释代码，既避免只讲逐行翻译，也避免扩展成过度审计。

## 通用原则

- 先给结论，再给关键证据。
- 事实只来自用户提供的代码、当前仓库检索结果或明确来源；不确定就说「不确定」。
- 涉及仓库代码时，尽量给出 `file:line` 证据。
- 区分「代码实际做了什么」和「我推测它为什么这样设计」。
- 不主动改代码；如果发现明显问题，只指出并说明影响。

## 场景 1：解释一段代码

按以下 2 到 3 个角度输出：

1. **一句话结论**：这段代码的目的、输入、输出或主要状态变化。
2. **核心流程**：按执行顺序讲关键步骤、分支和循环，不做逐行翻译。
3. **注意点**：只列最关键的边界条件、隐含假设或潜在风险；没有就写「未发现必须强调的风险」。

推荐输出结构：

```markdown
结论：...

关键逻辑：
1. ...
2. ...
3. ...

注意点：
- ...
```

## 场景 2：追踪一个函数

先定位定义和调用，再按以下 2 到 3 个角度输出：

1. **调用链**：入口、直接调用者、关键下游调用；只展开影响理解主流程的节点。
2. **核心对象与数据流**：函数读写了哪些对象、参数如何变形、结果交给谁消费。
3. **流程与原理**：解释设计意图、关键机制、为什么这样串起来；推测必须标注「推测」。

推荐输出结构：

```markdown
结论：...

证据：
- `path/to/file.py:10`：定义或入口
- `path/to/file.py:42`：关键调用

调用链：
1. A -> B：...
2. B -> C：...

核心对象与数据流：
- ...

原理说明：
- ...
```

## 提示词模板

用户可直接这样触发：

```text
使用 $code-reading-assistant。
场景：解释代码 / 追踪函数
材料：粘贴代码，或给出函数名与文件路径。
要求：只从 2 到 3 个关键角度输出；先给结论，再给证据；不确定的地方明确说不确定。
```

## 文档沉淀提示

如果分析结果包含可复用的项目规则、关键链路、排障经验或架构知识，结尾追加一句：

> 这次结论有可复用价值，是否要我沉淀到指定文档？如果需要，请给出文档路径或允许我建议位置。

没有明显可复用价值时，不要追加这句。
