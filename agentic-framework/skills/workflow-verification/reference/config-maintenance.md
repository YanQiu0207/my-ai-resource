# 配置维护模式操作手册

面向 AI：用户显式触发「初始化 / 刷新 verify 配置」时，如何生成、刷新、试运行并写入 `verify.config.json`。本模式是配置的**唯一写入路径**，代码任务全程只读。设计依据见 `docs/design-docs/workflow-verification/ai-maintained-config/spec.md`。

## 流程

```text
检查仓库证据
→ 生成或刷新 verify.config.json
→ 校验配置结构
→ 试运行检查
→ 弱化类变更经用户确认
→ 写入并纳入 Git
```

`verify.config.json` 纳入 Git，使配置变化可审查；`.verify/`（基线与报告）维持不入库。

## 仓库证据

只根据以下文件和目录推导检查项：

- 依赖与项目清单：`pyproject.toml`、`package.json`、`go.mod`、`Cargo.toml`、解决方案或工程文件。
- 锁文件：用于确认项目实际使用的包管理器。
- CI 配置：GitHub Actions、GitLab CI 或其他流水线中的现有命令。
- 构建入口：`Makefile`、`Taskfile.yml`、项目脚本和已有命令包装器。
- 测试目录及测试配置：例如 `tests/`、`pytest.ini`、`vitest.config.*`。
- Lint、格式化和类型检查配置。
- 项目 `AGENTS.md`、Spec 和 ADR 中已经明确的工程约束。

若证据冲突，优先级为：

1. 当前 CI 中成功使用的命令。
2. 项目清单和官方脚本提供的命令。
3. 项目文档明确要求的命令。
4. 根据目录结构推导的候选命令。

第 4 类命令必须试运行成功后才能写入配置。

## 生成规则

### 检查分类

沿用 `verify.config.example.json` 的三类命名：

- **B 类·交付门槛**：编译或构建、不访问真实外部资源的测试、项目已启用的 Lint 或类型检查、测试数量不得减少等增量保障。
- **A 类·静态规范**：禁用 API、语法或硬编码等 `forbid_pattern` 检查。只能来自 `AGENTS.md`、Spec、ADR、CI 或现有检查脚本，没有明确证据时不生成。
- **C 类·工程一致性**：项目已有的自定义校验脚本、既有指标不得恶化。

生成优先级：B → A → C。字段写法见 [config-guide.md](config-guide.md)，每项检查用 `_note` 说明目的和边界。

### 最低有效配置

生成的 `checks` 不能为空，且至少包含一项能成功执行的编译（构建）检查，或不访问外部资源的测试。

如果仓库没有可安全执行的编译或测试入口，返回 ERROR。不得用恒定返回 `0` 的占位命令伪造有效配置。

### 外部资源检查

命令涉及以下资源时，默认不进入常规门禁：

- 真实 API 或付费服务。
- 生产数据库、生产队列或云资源。
- 需要用户凭证的服务。
- 会修改外部状态的集成测试。

排除方式使用项目已有机制（如 pytest marker `-m "not integration"`、测试分组或独立测试目录），并在 `_note` 中说明。

## 刷新规则

配置已存在时，审查以下变化：

- 技术栈、依赖清单或包管理器变化。
- CI 新增、删除或修改验证命令。
- 新增测试框架、Lint、类型检查或构建入口。
- 配置中的路径、命令或工具已经失效。
- Spec、ADR 或 `AGENTS.md` 新增明确的工程红线。

没有上述证据时保持配置不变。格式差异、字段排序或文案润色不构成更新理由。

## 更新安全规则

### 自动应用

- 新增有仓库证据、且试运行成功的检查。
- 修复因文件迁移、命令改名导致的失效路径。
- 根据 CI 的明确变化同步等价命令。
- 为外部资源测试补充项目已有 marker 或 tag 排除条件（「扩大测试排除范围」的唯一例外）。

### 需用户确认

- 删除检查。
- 降低 `threshold`。
- 把 `baseline_aware` 从 `true` 改为 `false`。
- 扩大测试排除范围（外部资源排除除外）。
- 把失败检查改成恒定成功或追加吞错逻辑。

确认时必须展示「旧值、新值、仓库证据、修改理由」，用户同意后才写入。无法提供证据时不得提议修改。

## 试运行

写入前依次执行：

1. 校验 JSON 结构和字段（对照 [config-guide.md](config-guide.md) 的字段速查）。
2. 检查命令依赖是否可用。
3. 试运行所有非外部资源检查。
4. 修复由 AI 生成的错误命令。
5. 确认命令无破坏性副作用：只允许读取和向项目目录写构建产物；包含删除、网络写入或系统配置修改的命令一律排除。

试运行失败的处理：

- 命令推导错误：AI 修正后重试。
- 项目本身已有失败：适合增量比较时改为基线检查（`baseline_aware: true`）；不适合时报告 ERROR，不删除检查。
- 工具未安装：不自动安装，报告缺失依赖。
- 需要外部资源：从常规门禁排除，并在 `_note` 中说明。

## 输出模板

模式结束时输出：

```text
配置状态：新建 | 已更新 | 无变化 | 初始化失败
配置证据：<CI、项目清单、测试配置等路径>
新增检查：<列表>
修改检查：<旧值 → 新值及理由，含用户确认结果>
排除检查：<外部资源检查及理由>
试运行：PASS | ERROR
```

## 信任边界

生成的 `command` 会在本模式中被试运行、之后由 `verify.py` 经 Shell 执行。只在用户信任的仓库中进入本模式；审查外部 PR 或不可信分支时不得自动生成或试运行配置。
