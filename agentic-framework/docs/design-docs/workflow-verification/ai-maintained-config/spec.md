# AI 维护 Verify 配置

**状态**：Accepted

## 结论

`verify.config.json` 的生成与刷新由用户手动命令触发：用户运行配置命令后，AI 根据仓库证据生成或刷新配置，经试运行确认有效后写入。代码任务只消费配置，不修改配置：缺失时暂停并提示用户先运行配置命令，发现失效迹象时只在报告中提醒。

不采用「代码任务自动生成或刷新配置」。更新必须有仓库证据；删除检查、降低阈值等弱化类变更必须经用户确认。

## 问题

当前行为存在两个缺口：

- 没有 `verify.config.json` 时，只运行内置 Spec Drift，项目的编译、测试和 Lint 不会自动进入门禁。
- 配置需要人手工编写命令和维护，容易长期不更新，与项目实际命令脱节。

参考 `skills/workflow-verification/reference/verify.config.example.json`，有效配置通常同时包含：

- 必过检查：测试、编译。
- 增量检查：类型错误不得增加、测试数量不得减少。
- 项目约束：排除会访问真实 API 的集成测试。
- 可解释信息：每项检查通过 `_note` 说明目的和边界。

## 目标

- 配置由 AI 根据仓库现有事实生成，不要求用户手写命令；写入只发生在用户显式触发的配置命令中。
- 代码任务缺少配置时不静默降级为只剩 Spec Drift，暂停并引导用户初始化。
- 刷新只在有仓库证据时更新，无证据时配置保持字节级不变。
- 新配置必须经过实际试运行，不能只生成 JSON。
- 配置在代码任务实现期间不能被修改或弱化。
- 弱化类变更必须经用户确认。
- 配置命令与代码任务的报告都说明配置状态。

## 非目标

- 不让 AI 凭空制定项目规范。
- 不自动安装新的构建、测试或 Lint 工具。
- 不要求一次性清理项目的全部历史违规。
- 不执行需要真实凭证、付费 API、生产资源或破坏性操作的检查。
- 不用自动生成替代项目 CI；Verify 只消费已有工程能力。

## 触发时机

### 配置命令（唯一写入路径）

用户显式要求「初始化 / 刷新 verify 配置」（或运行薄包装命令，如 `/verify-config`）时，`workflow-verification` 进入配置维护模式：

```text
检查仓库证据
→ 生成或刷新 verify.config.json
→ 校验配置结构
→ 试运行检查
→ 弱化类变更经用户确认
→ 写入并纳入 Git
```

配置文件纳入 Git，使配置变化可审查、可追踪。

### 代码工作流（只读）

`workflow-code-generation` 在 Fast-Path / Phase 0 动代码前：

- 配置存在 → 采集 `.verify/baseline.json` 并冻结配置，正常进入实现。
- 配置缺失 → 暂停，提示用户先运行配置命令；用户明确选择跳过时，按现状只跑内置门禁并在报告标注。
- 验证过程中出现 exit 2，或配置引用的命令、路径失效 → 报告提醒用户之后运行配置命令刷新，不阻断当前任务。

代码任务全程不修改 `verify.config.json`。

### 用户主动验证

用户只要求「运行 Verify」时：

- 配置存在：直接执行，不改写配置。
- 配置缺失：报告 ERROR，提示先运行配置命令。

## 仓库证据

AI 只根据以下文件和目录推导检查项：

- 依赖与项目清单：`pyproject.toml`、`package.json`、`go.mod`、`Cargo.toml`、解决方案或工程文件。
- 锁文件：用于确认项目实际使用的包管理器。
- CI 配置：GitHub Actions、GitLab CI 或其他流水线中的现有命令。
- 构建入口：`Makefile`、`Taskfile.yml`、项目脚本和已有命令包装器。
- 测试目录及测试配置：例如 `tests/`、`pytest.ini`、`vitest.config.*`。
- Lint、格式化和类型检查配置。
- 项目 `AGENTS.md`、Spec 和 ADR 中已经明确的工程约束。

若证据冲突，优先级为：

1. 当前 CI 中成功使用的命令。
2. 项目清单和官方脚本提供的命令。
3. 项目文档明确要求的命令。
4. 根据目录结构推导的候选命令。

第 4 类命令必须试运行成功后才能写入配置。

## 生成规则

### 检查分类

沿用 `verify.config.example.json` 的三类命名：

- **B 类·交付门槛**：编译或构建、不访问真实外部资源的测试、项目已启用的 Lint 或类型检查、测试数量不得减少等增量保障。
- **A 类·静态规范**：禁用 API、语法或硬编码等 `forbid_pattern` 检查。只能来自 `AGENTS.md`、Spec、ADR、CI 或现有检查脚本，没有明确证据时不生成。
- **C 类·工程一致性**：项目已有的自定义校验脚本、既有指标不得恶化。

生成优先级：B → A → C。

### 最低有效配置

生成的 `checks` 不能为空，且至少包含一项能成功执行的编译（构建）检查，或不访问外部资源的测试。

如果仓库没有可安全执行的编译或测试入口，配置命令返回 ERROR。不得用恒定返回 `0` 的占位命令伪造有效配置。

### 外部资源检查

命令涉及以下资源时，默认不进入常规门禁：

- 真实 API 或付费服务。
- 生产数据库、生产队列或云资源。
- 需要用户凭证的服务。
- 会修改外部状态的集成测试。

排除方式使用项目已有机制（如 pytest marker `-m "not integration"`、测试分组或独立测试目录），并在 `_note` 中说明。

## 刷新规则

配置命令执行时审查以下变化：

- 技术栈、依赖清单或包管理器变化。
- CI 新增、删除或修改验证命令。
- 新增测试框架、Lint、类型检查或构建入口。
- 配置中的路径、命令或工具已经失效。
- Spec、ADR 或 `AGENTS.md` 新增明确的工程红线。

没有上述证据时保持配置不变。格式差异、字段排序或文案润色不构成更新理由。

## 更新安全规则

### 自动应用

- 新增有仓库证据、且试运行成功的检查。
- 修复因文件迁移、命令改名导致的失效路径。
- 根据 CI 的明确变化同步等价命令。
- 为外部资源测试补充项目已有 marker 或 tag 排除条件（「扩大测试排除范围」的唯一例外）。

### 需用户确认

- 删除检查。
- 降低 `threshold`。
- 把 `baseline_aware` 从 `true` 改为 `false`。
- 扩大测试排除范围（外部资源排除除外）。
- 把失败检查改成恒定成功或追加吞错逻辑。

确认时必须展示「旧值、新值、仓库证据、修改理由」，用户同意后才写入；交付报告同步记录。无法提供证据时不得提议修改。

### 实现期间冻结

代码任务采集基线后，本次任务不得修改 `verify.config.json`。实现确实引入新的构建或测试入口时：

- 不修改配置，按现有配置完成验证。
- 在交付报告「Verify 配置」部分标注新入口，提示用户运行配置命令后纳入门禁。

不得在验证失败后直接重新采集基线。

## 试运行

配置命令写入前依次执行：

1. 校验 JSON 结构和字段。
2. 检查命令依赖是否可用。
3. 试运行所有非外部资源检查。
4. 修复由 AI 生成的错误命令。
5. 确认命令无破坏性副作用：只允许读取和向项目目录写构建产物；包含删除、网络写入或系统配置修改的命令一律排除。

试运行失败的处理：

- 命令推导错误：AI 修正后重试。
- 项目本身已有失败：适合增量比较时改为基线检查；不适合时报告 ERROR，不删除检查。
- 工具未安装：不自动安装，报告缺失依赖。
- 需要外部资源：从常规门禁排除，并在 `_note` 中说明。

## 输出要求

配置命令结束时输出：

```text
配置状态：新建 | 已更新 | 无变化 | 初始化失败
配置证据：<CI、项目清单、测试配置等路径>
新增检查：<列表>
修改检查：<旧值 → 新值及理由，含用户确认结果>
排除检查：<外部资源检查及理由>
试运行：PASS | ERROR
```

代码任务的最终报告只说明消费情况：

```text
Verify 配置：使用现有配置 | 缺失（已暂停提示 / 用户选择跳过） | 建议刷新（原因）
基线：<路径>
```

## 与现有机制的关系

- `workflow-verification` 新增配置维护模式，负责证据审查、生成规则、试运行和写入；仅由用户显式触发。
- `workflow-code-generation` 在 Fast-Path / Phase 0 只读检查配置并采集基线，缺失时暂停提示。
- `verify.py` 继续负责确定性执行、结构化报告、配置快照和退出码，不负责调用 AI。
- 检查分类沿用 `reference/verify.config.example.json` 的 A / B / C 命名。
- ADR 001 决策 2（验证时不猜测项目命令）保持不变：命令推导只发生在用户显式触发的配置命令中，用户在场且所有命令经试运行，不属于该 ADR 放弃的「自动猜测并运行项目命令」。
- `verify.config.json` 纳入 Git；`.verify/`（基线与报告）维持不入库。

## 验收标准

### 缺少配置

- 给定一个已有单元测试的 Python 项目且没有 `verify.config.json`。
- 进入代码工作流后暂停并提示运行配置命令。
- 用户运行配置命令后，AI 生成非空配置，至少包含可执行的测试或编译检查。
- 所有命令经过试运行后才写入。

### 配置刷新

- 给定一个已经配置 pytest 的项目。
- CI 新增 mypy 后，用户运行配置命令时识别该变化并增加对应检查。
- 仓库没有相关变化时，配置保持字节级不变。

### 防止弱化

- 代码任务实现期间 `verify.config.json` 被修改时，验证按配置快照失败关闭。
- 删除检查、降低阈值或关闭 `baseline_aware` 只能在配置命令中发生，且必须经用户确认。
- 验证失败后重新采集基线不能作为修复方式。

### 外部资源边界

- 项目存在调用真实 API 的集成测试时，常规门禁不会执行该测试组。
- 排除方式来自项目已有 marker、测试分组或测试配置，并写入 `_note`。

### 无法初始化

- 仓库没有安全、可执行的编译或测试入口时，配置命令返回 ERROR。
- 报告说明缺少什么，不生成空 `checks` 或占位命令。

## 信任边界

配置中的 `command` 由 `verify.py` 经 Shell 执行。配置命令只在用户信任的仓库中运行；审查外部 PR 或不可信分支时，不得自动生成或试运行配置，沿用 SKILL.md 的人工确认要求。
