# 项目知识沉淀规范

## 核心哲学：只沉淀代码读不出的东西

本框架对「系统真相」分三层，沉淀策略各不相同：

| 真相 | 谁是权威 | 怎么对待 |
|------|---------|---------|
| **现状**（系统现在怎么做：模块、接口、数据流） | **代码** | **不维护现状文档**。每次需要现状，重新分析代码提取（这是 `workflow-*` 设计工作流的既定动作）。任何手写的「系统现状」文档都会和代码漂移。 |
| **架构快照**（模块边界、调用链、设计模式的时效性快照） | **代码 + 快照文件共同维护** | **按需落地，显式标注时效性**。由代码探索或 Q&A 产出后写入；git commit 变更后结构类快照自动失效，洞察类需人工判断。编码时仍以代码为准，快照仅为导航工具。 |
| **intent**（当初为什么这么做、放弃了什么、什么约束、什么权衡） | **文档** | **必须沉淀**。代码里根本读不出 intent，不沉淀就永久丢失。载体是 **ADR** + 已归档的 feature spec。 |

> 这是本规范与 OpenSpec 式「中央行为规范库」的根本分歧：我们不维护会和代码争真相的「现状规范库」，只沉淀代码无法还原的 intent。架构快照是显式的受控例外——它有时效性标注，且编码时永远以代码为准。

## 加载时机

- 创建或更新 `docs/design-docs/**/spec.md`、`tasks.md` 时加载
- **每次向用户宣布完成 / 交付前加载**（触发交付前沉淀检查；归档 spec 只是其中一种场景）
- 改动涉及不可逆/高影响的架构决策、重大技术选型、放弃某方案时加载
- 用户询问 ADR、spec 归档或文档应放在哪里时加载
- 落地 arch-snapshot 前（由 codebase-researcher 或主对话触发写入时）加载
- 查证完一个故障根因、需要沉淀踩坑经验时加载

## 目录结构约定

```
docs/
├── adr/                             # 架构决策记录 —— intent 沉淀的主载体
│   └── NNN-<title>.md               # 例：001-choose-message-queue.md
├── arch-snapshots/                  # 架构快照 —— 时效性导航工具
│   └── <module>/
│       ├── structure.md             # 结构 track：模块边界、接口、数据流
│       └── insights.md              # 洞察 track：设计模式、关键权衡
├── design-docs/                     # Feature Spec（按功能创建）
│   └── <module>/
│       └── <feature>/
│           ├── spec.md              # 需求 + 设计 + 为什么（intent 快照）
│           └── tasks.md             # 任务拆分与进度
└── issues/                          # 已查证的故障与踩坑
    └── <issue>.md
```

> 不设 `architecture/` 现状散文目录：系统现状以代码为准，由设计工作流每次重新调研，不靠手写文档保鲜。`arch-snapshots/` 是显式例外，以 git commit hash 标注时效性。

## 各类文档的定位与职责

### Feature Spec（`docs/design-docs/`）

每个功能或较大改动对应一个子目录，生命周期跟随功能：

```
Draft → In Review → Approved → Archived
Quick Draft → Approved → Archived
```

- **Spec**：记录「为什么做、做什么、怎么做」。它的「现状分析」部分会随代码过期（无妨，下次重读代码）；它的**「为什么/目标/备选方案/权衡」是 intent 快照，有持久价值**。
- **Tasks**：记录「分几步做，每步的完成条件是什么」，开发过程中实时更新。
- **归档**：先过[交付前沉淀检查](#交付前沉淀检查强制)，再把 spec.md 头部的 `状态` 字段改为 `Archived`，**文件原地保留，不移动**。
  - 原地保留的好处：避免移动文件导致已有引用失效，且按 `<module>/<feature>` 目录天然可检索。

### 架构快照（`docs/arch-snapshots/`）

代码探索或 Q&A 产出架构级信息后，可落地为时效性快照，供快速定向使用。**编码时仍以代码为准。**

**两种 track：**

- **structure track**（`structure.md`）：模块边界、关键接口、数据流、调用链。事实类，git commit 变更后自动失效。
- **insights track**（`insights.md`）：设计模式、关键权衡、「为什么这样设计」。长期有价值；若涉及不可逆技术选型或影响多模块的约束，直接写 ADR，不经 insights track 中转。

**frontmatter 格式（必填）：**

```yaml
---
module: <module>
vcs_ref: git:<hash> | svn:<revision> | N/A
generated_at: <YYYY-MM-DD>
track: structure | insights
expires_hint: "structure track 在版本库提交后自动失效；insights track 需人工判断"
---
```

**vcs_ref 获取方式：**
- git 仓库：`git rev-parse HEAD`，写为 `git:<hash>`（如 `git:a3f2c1b`）
- SVN 工作区：`svn info --show-item last-changed-revision`，写为 `svn:<revision>`（如 `svn:1234`）
- 无版本控制：写 `N/A`

**Overlap 检测**：落地前先检查同模块是否已有同 track 文件。
- 已有且 `vcs_ref` 与当前版本库一致 → 自动跳过，不询问
- 已有但 `vcs_ref` 已过期或为 `N/A` → 询问用户「更新 / 跳过」
- 无 → 直接写入

**何时写快照：**
- 探索或 Q&A 结束后，收集到模块级信息（模块边界 / 关键接口 / 重要设计模式 / 调用链）
- 探索范围跨越单个文件，或产出了跨文件调用关系 / 模块级概览（纯单文件内查询跳过）
- 由 codebase-researcher 或主对话在产出末尾询问用户确认，不自动写入

**碎片化知识点的归属：**
- 属于「这个类/函数做什么、在哪里」→ structure track
- 属于「为什么这样设计、有什么权衡」→ insights track；若重要性足够，直接写 ADR

### 架构决策记录（`docs/adr/`）—— intent 沉淀的主载体

记录**代码读不出、且不可逆或高影响的决策**，一旦写定不修改（只更新状态字段）：

```markdown
# NNN: <决策标题>

**状态**: Accepted / Superseded by NNN

## 背景
<!-- 当时面临什么问题，为什么需要做这个决定 -->

## 决策
<!-- 我们选择了什么方案 -->

## 放弃的方案
<!-- 考虑过但排除的选项，以及排除原因 -->

## 后果
<!-- 这个决定带来了什么，有什么已知的权衡 -->

## 适用条件（可选）
<!-- 这个决策在什么前提下成立；条件变化后需重新评估 -->
```

**何时写 ADR：**
- 引入/放弃某个存储系统、消息队列、框架或技术方向
- 选择某个有显著权衡的架构模式（且权衡理由代码里看不出来）
- 确立团队编码约定
- 引入一条新的红线约束（性能/兼容/安全），其「为什么」需要被后人理解

**触发时机**：做出不可逆/高影响决策时**即时询问**是否写 ADR，不等到交付前——上下文冷却后写出来的 ADR 质量显著下降。交付前沉淀检查作为补网，不是主触发点。

### 故障与踩坑（`docs/issues/`）

记录**已查证的故障和开发坑点**。根因属于代码读不出的经验：代码只展示修复后的样子，读不出哪里坏过、为什么坏、什么场景会复现。

**收录门槛**：根因经过验证——有证据链，修复后回归通过。未验证的猜测不收，错误的排障结论比没有更危险。

**条目骨架**：

```markdown
# <issue 标题>

## 现象
## 根因
## 修复
## 适用边界
<!-- 什么条件下这条经验成立；环境/版本变化后需重新验证 -->
## 证据
<!-- 复现步骤、日志、commit、验证记录 -->
```

**与其他载体的边界**：

- 故障引出了不可逆决策（如更换组件、立红线）→ 另写 ADR，issue 与 ADR 互链
- 属于设计权衡而非故障 → arch-snapshots 的 insights track
- 与本项目无关、跨项目仍成立 → 写入共用知识库（见交付前沉淀检查）

## 交付前沉淀检查（强制）

> **每次向用户宣布「完成 / 交付」前**（无论是否归档 spec），**必须**执行本检查，并**逐条给出判定结论**——不得跳过、不得默认略过。归档 spec 只是其中一种触发场景。

### 检查什么

逐条自问，判断本次变更里有没有「代码读不出、值得沉淀」的 intent：

| 信号 | 若命中 → 动作 |
|------|--------------|
| 做了不可逆 / 高影响的架构决策 | 写 / 更新一条 ADR |
| 放弃了某个方案，且放弃理由重要 | 写 ADR，或确认 spec 的「备选方案/权衡」章节已写清 |
| 引入了新的红线约束（性能/兼容/安全） | 写 ADR 或在 spec 中明确记录约束与理由 |
| 查证了一个故障根因，或踩了值得记录的坑 | 写 / 更新 `docs/issues/` 条目（仅收已验证的根因） |
| 产生了与本项目无关、可跨项目复用的结论 | 写入共用知识库 `domains/`（AGENTS.md 已接线时），项目内留引用 |
| 只是普通功能/配置/bugfix，代码自解释 | 显式判定「无需沉淀」 |

### 判定规则

1. **现状类信息不沉淀**：模块怎么变、接口长什么样、数据流如何——这些是代码的职责，不写进文档。
2. **必须逐条作答**：对照上面的信号表，每个信号逐行给出「命中 → 已沉淀到 <具体 ADR/spec 章节>」或「未命中」。单句笼统的「判定无需沉淀」不构成有效判定；沉默 = 未完成。
3. **文档债务**：若判定需要沉淀但当前无法完成，在 `tasks.md` 建一个「intent 沉淀」任务，不要跳过。

## 文档与代码的同步原则

| 变更类型 | 需要做的 |
|---------|---------|
| 功能开发完成 / 准备归档 | **执行交付前沉淀检查**；通过后 spec 状态改为 `Archived` |
| 不可逆/高影响决策、放弃方案、新红线约束 | **即时**写 / 更新 ADR（不等交付前） |
| 查证了故障根因 / 踩了值得记录的坑 | 写 / 更新 `docs/issues/` 条目 |
| 系统现状变了（模块/接口/数据流） | **不写现状文档**——现状以代码为准，下次重新调研；已有 arch-snapshot 中的 structure track 自动失效 |
| 代码探索或 Q&A 产出了模块级理解 | 询问用户是否落地到 `docs/arch-snapshots/<module>/`（structure / insights track） |
| 编码规范变更 | 对应 `std-*` skill 文件 |
| 工程流程变更 | 对应 `workflow-*` skill 文件 |

## 与其他工作流的关系

| 工作流 | 产出 | 在哪里 |
|-------|------|--------|
| `workflow-requirements-clarification` | spec.md（1-3 章：背景/目标/需求） | `docs/design-docs/<module>/<feature>/spec.md` |
| `workflow-system-design` | spec.md（4-8 章：设计/备选/权衡） | 同上 |
| `workflow-quick-design` | spec.md（Quick Draft） | 同上 |
| `workflow-code-generation` | tasks.md + 代码 + **交付前沉淀检查** | `docs/design-docs/<module>/<feature>/tasks.md` |
| `self-refinement` | skill 文件更新 | `skills/` 目录下对应 skill |

ADR 不由设计工作流自动创建，而是在**交付前沉淀检查**判定「值得沉淀」时显式创建；若交付时无法立即完成，在 `tasks.md` 中保留为未完成任务。
