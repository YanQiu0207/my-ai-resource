# 前端 Playwright 验证

## 核心规则

验证运行中的页面，不只看源码。使用浏览器自动化走真实用户路径，用 DOM 状态验证功能正确性，用截图验证视觉正确性。验证失败时，修复问题并重复验证循环，直到验收标准通过，或证明存在具体阻塞。

## 工作流程

1. 编辑前明确验收目标：页面、路由、用户角色、视口、交互步骤、预期 DOM 状态和预期视觉效果。
2. 按宿主项目约定修改代码、配置或模板。
3. 让改动生效：
   - 仅静态资源改动：按项目需要刷新或重新构建。
   - 服务端模板、路由、后端逻辑、运行时配置或 API 改动：按项目 runbook 重启受影响服务。
4. 用项目健康检查或目标 URL 证明应用可访问。
5. 使用 Playwright 或可用的浏览器 MCP 打开浏览器。
6. 按真实用户方式登录。优先使用项目说明中的账号；没有说明时，选择已记录的本地/开发测试账号。
7. 通过真实 UI 路径导航：点击、填写表单、切换 Tab、筛选、打开菜单和链接。若被测行为依赖导航状态，不要直接跳到最终 URL。
8. 检查 DOM 事实：
   - URL 和路由正确。
   - 必要文案、标签、控件、行数据、状态标记、错误态和空态存在。
   - 表单值、启用/禁用状态、选中 Tab、弹窗可见性、动态/局部更新结果符合预期。
   - 关键按钮或链接可见且可操作。
9. 截图做视觉检查：
   - 可见 UI 改动至少保留一个桌面视口截图。
   - 涉及响应式布局时补移动端或窄屏视口。
   - 失败时保留失败截图；必要时同时保留页面 HTML 或 DOM 快照。
10. 结合 DOM 和截图分析。DOM 证明行为与内容正确；截图证明布局、间距、溢出、遮挡、可读性和视觉层级正确。
11. 如发现异常，根据浏览器证据、日志和源码定位问题，然后从“让改动生效”步骤重新验证。
12. 交付时说明真实用户路径、视口、DOM 检查、截图位置、服务重启/构建动作，以及未验证风险。

## DOM 与截图分工

用 DOM 检查确定性事实：登录成功、元素存在、中文文案、表格行数、按钮启用状态、弹窗打开/关闭、状态流转、表单提交结果，以及 HTTP 驱动的局部更新。

用截图检查设计事实：重叠、裁剪、留白、层级、密度、上下文中的颜色对比、响应式换行、长文本表现、媒体渲染，以及整体是否像一个成熟的页面。

不要把截图单独当作流程可用的证明。不要把 DOM 文本单独当作视觉效果可接受的证明。

## 截图归档

- 若项目存在 `docs/design-docs/<module>/<feature>/`，将验证截图保存到该目录下的 `screenshots/`。
- 若没有对应设计目录，保存到本次任务的临时验证目录，并在交付说明中写明路径。
- 文件名体现页面、视口和状态，例如 `issue-detail-desktop.png`、`issue-detail-mobile-error.png`。

## 交互标准

优先使用可访问定位器和用户可见文本。只有应用已有稳定 test id，或本次需求明确要添加 test id 时，才依赖 test id。每次操作后等待可观察的 UI 状态，不依赖固定 sleep。

涉及破坏性动作时，使用种子数据、本地数据或可回滚路径。若流程可能修改重要实时数据且没有安全夹具，停止并说明阻塞，不要盲目模拟操作。

## 视觉检查清单

每次可见 UI 改动交付前检查：

- 文本没有重叠、裁剪、意外横向滚动或不可读换行。
- 主要控件可见、可点击，未被 sticky 头部/底部遮挡。
- 相关时考虑加载态、空态、错误态、长文本和高密度数据状态。
- 桌面和窄屏视口都能完成核心任务。
- 用户可见文案符合项目语言和语气规则。

## 交付格式

最终回复包含：

- 通过浏览器验证了什么。
- 使用的登录用户和用户路径。
- 执行的 DOM 检查。
- 截图路径，或说明截图只通过工具检查未落盘。
- 服务重启或构建动作。
- 无法验证的剩余风险。
