# 002：保留 Fast-Path 并扩大适用范围

**状态**：Accepted

## 背景

排查「简单改动执行耗时」问题时，出现了删除 Fast-Path、所有代码修改统一交给 agent 执行的提议，理由是 Fast-Path 引用扩散带来维护成本、路由判断与外溢切换带来状态复杂度。

同时发现两套规则分开手写已产生实际漂移：Fast-Path 的 review 修复循环写成「直到通过」（无轮次上限），标准流程同位置是「≤ 2 轮」。这证明双轨规则文本的漂移风险是真实的。

原始 Fast-Path 判据（单文件、局部、改一行级）过窄，大量低风险小改动被迫走完整标准流程。

## 决策

1. 保留 Fast-Path，执行主体仍为主会话直改，不起 agent / worktree。
2. 扩大判据，主判据为「**请求即计划**」：用户请求本身已完整确定改什么、怎么改，AI 无需做任何未言明的设计决策。同时须满足：路由阶段可确定完整文件列表、修改局部（不改函数签名 / 模块边界 / 公开接口）、不碰数据 / 权限 / 并发 / 安全 / 性能关键路径。文件数只作护栏（默认 ≤ 3 个，同一模式的机械重复除外），不作主判据——文件数与风险仅弱相关，真正过滤风险的是上述定性判据。机械重复是指每处应用相同变换，不改变接口、契约、控制流或模块交互。
3. Fast-Path 固定使用 `lightweight` 档，只派一个 `comprehensive-reviewer`，但该 reviewer 同时覆盖工程规范、需求符合度、正确性与健壮性。速度优化来自减少编排和重复派发，不减少质量检查维度；发现性能、并发、跨模块契约变化或信任链等高风险线索时，退出 Fast-Path。
4. Fast-Path 与标准流程共用同一套质量门规则文本（review 循环语义、验证、沉淀检查、交付门），Fast-Path 只是参数不同（执行主体 = 主会话、免 spec / tasks、`review_profile: lightweight`），避免再次漂移。
5. 外溢安全网不变：执行中发现超出路由判据 → 立即退出转标准流程。

## 放弃的方案

### 删除 Fast-Path，统一 agent 执行

放弃原因：不解决「简单改动耗时」——反而给最轻的改动增加 dispatch + worktree + merge 的固定开销，方向与问题相反。其宣称的「消除分支」也不成立：只要极轻改动仍免完整 spec / tasks，路由判断和特例分支就仍然存在，只是执行主体换成 agent；外溢处理还从「主会话就地转标准流程」变成「agent 中止、回报、主会话重启」，状态复杂度不降反升。功能性引用实测仅 4 处（skill 两处、脚本注释、README 注释），维护面被高估。

### 彻底消除分支（轻量改动也走完整 spec / tasks / 用户批准）

放弃原因：一行级改动也要 spec lint、tasks.md、用户批准闸口，小改动成本剧增，与问题背景直接冲突。

## 后果

- Fast-Path 覆盖面扩大后，「测试与实现同批交付（Fast-Path 除外）」的免测试豁免面同步扩大；风险由 lightweight review + 机器验证兜底，是否对 Fast-Path 内的新增逻辑补测试要求，留待观察后另行决策。
- Fast-Path 固定使用单个综合 reviewer，避免多 reviewer 派发成本；正确性与健壮性仍在首轮审查范围内，不以降低质量覆盖换取速度。
- 本决策修订 arch-snapshots `insights.md` 取舍 3 的适用范围描述（选择本身维持）；快照按 vcs_ref 自然过期，以本 ADR 为准。

## 适用条件

- 适用于 `workflow-code-generation` 的复杂度路由。
- 生产关键系统仍按框架边界声明回到逐 task 人工确认，不适用 Fast-Path。
