# 为什么要这个框架 + 方法论

## 1. 问题：`workflow-*` 适合生产单变更，不适合快速搭大型非生产工具

原框架的 `workflow-*` 是为「**生产项目里的单个变更**」设计的：

- 逐 task 停等用户批准（每个 task 完成都停下）
- 每 task 按风险分级 code review（小需求轻量审，高风险全量审）
- verify 基线门禁

这套用「慢」换「零缺陷」，在生产环境改一处代码时是对的。但从 0 快速搭一个**大型非生产工具**（功能多、复杂、但不上生产）时，光门禁就把「快」杀光了。

反过来，纯 vibe coding 做大工具又会失控：功能一多、抽象一乱，就是返工地狱。

**核心矛盾**：大工具的复杂度**需要结构**，但生产级门禁**太慢**。方法论的本质是在两者间找平衡，而不是二选一。

## 2. 诉求画像：前重后自主

这套框架服务的开发者诉求是明确的「**前重后自主**」：

| 阶段 | 诉求 |
| --- | --- |
| **前期** | 愿意花精力聊清需求、审核设计 → 重设计门 **OK** |
| **执行** | 要 AI **自主 + 并行 + 高质量**；**不要人逐步干预** |
| **质量 vs 成本** | 质量优先，但按风险分级投入 token：小需求轻量审，高风险再多 reviewer / TDD |

一句话：人在设计阶段把关，执行阶段交给 AI 又快又好地跑完。

## 3. 方法论：保留前后两道人/机门，自主化中间执行

```
┌─ 前期（人把关，保留原 workflow-*）─┐   ┌─ 执行（AI 自主并行，新增）─┐   ┌─ 收尾（人把关）─┐
│ requirements-clarification        │   │ 并行 dispatch implementer │   │ 一次性汇总       │
│ system-design / quick-design      │ → │ 每产物过 review + verify  │ → │ intent 沉淀检查   │
│ 任务拆解 + 用户批准 tasks.md       │   │ worktree 隔离 · 不停等     │   │ 交付            │
└───────────────────────────────────┘   └───────────────────────────┘   └─────────────────┘
```

三段的设计原则：

1. **前期设计门保留**（人愿意付出）—— `requirements-clarification` + `system-design`/`quick-design` 不变。设计 + 任务清单经用户批准，是人把关的最后一道闸。
2. **执行段自主化**（新增的核心）—— 任务清单批准后，**并行**派 implementer subagent 实现，每个产物照样按风险过 `code-review` + `verify.py`，但**不再逐 task 停等用户**，全部跑完一次性汇报。质量靠 review/verify 兜底，而非人逐步盯。
3. **收尾人把关** —— 全部完成后一次性汇报 + 按 `project-knowledge` 做交付前 intent 沉淀检查。

> 可选实践：动手前先让一条最核心的端到端链路跑通，验证核心抽象对不对，再批量并行铺开其余功能。属前期设计阶段的事，按需采用，不强制。

## 4. 这套相对 `workflow-*` 改了什么

只改**执行段**：把「逐 task 停等的串行执行」换成「下放 agent 执行（单 agent / 多 task 并行），批准后自主连跑」，并入 `workflow-code-generation` 的复杂度路由（极轻改动仍主会话直改）。前期设计门、质量基建（多维 review、verify、intent 沉淀）**全部保留**。详见 [03-parallel-execution-mode.md](03-parallel-execution-mode.md)。
