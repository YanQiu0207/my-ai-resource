# 001：将机器验证设为合并前硬门

**状态**：Accepted

## 背景

LLM Code Review 擅长判断设计、可读性和潜在风险，但不能替代编译、测试、Lint 等可重复执行的客观检查。如果只依赖 LLM 宣称「应该没问题」，代码可能在未通过基础工程检查时被合并。

同时，存量项目可能已经存在历史违规。若机器门禁要求一次性清零全部历史问题，接入成本过高；若完全不检查，又无法阻止本次变更新增问题。代码变更还可能与 Spec、Tasks 或 ADR 脱节，使实现意图无法追溯。

## 决策

1. 将 `workflow-verification` 设为与 LLM Code Review 并列的合并前硬门。Review 通过不代表可以合并，机器验证也必须通过。
2. 使用项目根目录的 `verify.config.json` 声明编译、测试、Lint 和自定义检查。没有配置时不猜测项目命令，只执行内置门禁并明确报告未运行项目自定义检查。
3. 对适合基线比较的检查，只阻止本次变更新增违规，不要求当前任务清理全部历史债务。
4. 在改动前单点采集基线；并行 Worktree 只读同一份基线，分别生成验证报告，避免并发覆盖基线。
5. 验证时检查配置快照。删除、修改或关闭既有检查时失败关闭，防止通过弱化门禁获得假 PASS；新增检查按绝对模式执行。
6. 始终执行 Spec Drift 检查。代码发生变化时，必须满足以下任一条件：
    - 更新了能与代码变更机械关联的 `spec.md`、`ui-spec.md`、`tasks.md` 或 ADR；
    - 通过 `--spec-drift-reason` 提供具体、可审计的无需更新理由。
7. 标准流程在改动前记录 `base_sha`，后续显式传入 `--diff-base <base_sha>`。禁止在代码已经提交或合并后，使用默认 `HEAD` 掩盖本次变更。
8. 使用不同退出码区分结果：`0` 表示通过，`1` 表示代码或 Spec Drift 违规，`2` 表示配置、基线或执行环境导致门禁自身失效。

具体命令、配置字段和报告格式以 `skills/workflow-verification/SKILL.md` 及其 `reference/config-guide.md` 为执行真相源，本 ADR 只记录决策与权衡。

## 放弃的方案

### 只依赖 LLM Code Review

放弃原因：LLM 结论不可替代编译器、测试框架和静态检查工具的实际执行结果，也无法稳定产出可复核的退出码与结构化报告。

### 自动猜测并运行项目命令

放弃原因：不同项目的构建与测试入口差异较大，自动探测容易漏检、误执行或运行不受信任的命令。显式配置的行为更可审计。

### 要求存量违规一次性清零

放弃原因：会让已有项目难以接入门禁，并把当前变更与历史治理绑定。基线模式允许历史问题暂存，但禁止继续新增。

### 代码变更时强制更新任意规格文档

放弃原因：为了过门而制造无意义文档改动会降低文档可信度。允许提供具体理由，但空泛的「无需更新」不应作为有效证据。

## 后果

- 合并条件更严格：Review 与 Verify 任一失败都不能视为完成。
- 项目需要维护可信的 `verify.config.json`；不可信仓库中的命令不能直接自动执行。
- 基线必须在改动前采集。改动后重新采集可能把新增违规固化为历史问题，因此不能作为修复手段。
- Spec Drift 只能机械判断有限的路径关联，无法证明相关性时需要人工写明原因。
- `.verify/report.json` 成为本次机器验证的交付证据，但它属于本地临时产物，不纳入版本控制。

## 适用条件

- 适用于通过 `workflow-code-generation` 实施的代码变更，以及用户明确要求执行机器验证的场景。
- 纯文档变更不触发代码 Spec Drift 失败，但仍可运行配置中的项目检查。
- 外部 PR 或不可信分支必须先人工检查 `verify.config.json`，因为其中的命令会由 Shell 执行。
