# 为什么要这个框架 + 方法论

## 1. 问题：`workflow-*` 适合生产单变更，不适合快速搭大型非生产工具

原框架的 `workflow-*` 是为「**生产项目里的单个变更**」设计的：

- 逐 task 停等用户批准（每个 task 完成都停下）
- 每 task 按风险分级 code review（小需求轻量审，高风险全量审）

这套用「慢」换「零缺陷」，在生产环境改一处代码时是对的。但从 0 快速搭一个**大型非生产工具**（功能多、复杂、但不上生产）时，光门禁就把「快」杀光了。

反过来，纯 vibe coding 做大工具又会失控：功能一多、抽象一乱，就是返工地狱。

**核心矛盾**：大工具的复杂度**需要结构**，但生产级门禁**太慢**。方法论的本质是在两者间找平衡，而不是二选一。

## 2. 诉求画像：前重后自主

这套框架服务的开发者诉求是明确的「**前重后自主**」：

| 阶段 | 诉求 |
| --- | --- |
| **前期** | 愿意花精力聊清需求、审核设计 → 重设计门 **OK** |
| **执行** | 要 AI **自主 + 并行 + 高质量**；**不要人逐步干预** |
| **质量 vs 成本** | 质量优先，但按风险分级投入 token：小需求轻量审，高风险再多 reviewer / TDD |

一句话：人在设计阶段把关，执行阶段交给 AI 又快又好地跑完。

## 3. 底层推导：这些机制为什么会从 LLM 特性中自然产生

这套框架不是把传统研发流程机械套到 AI 上，而是接受一个前提：

> LLM 是强大的语义推理与生成器，但不是事务处理器、证明系统、持久状态机或可靠调度器。

框架的每项机制都应能从 LLM 的能力边界推导出来。

### 3.1 自然语言约束是软约束，机器状态才是硬约束

LLM 可以理解「先 Review、再 Verify、最后合并」，但理解不等于每次执行都严格遵守。长任务中出现新信息、工具失败或上下文切换后，模型可能重新解释原计划。

因此需要分离两类工作：

```text
语义工作：需求理解、设计、编码、问题分析 → 交给 LLM
确定性工作：依赖、状态、重试、阻塞、门禁 → 交给脚本
```

这直接导出：

- `spec.md` 和 `tasks.md` 把自然语言意图变成显式合同。
- `workflow_control.py` 负责 DAG、状态迁移、重试和恢复。
- `lint_spec.py`、`lint_task_deps.py`、`check_delivery.py` 把「应该遵守」变成机器退出码。
- Review 负责语义正确性，Verify 负责可执行事实；二者不能互相替代。

OpenAI Agents SDK 也明确区分「由 LLM 决定执行路径」和「由代码决定执行路径」两类编排方式，并允许混合使用。本框架的选择是：开放式问题交给模型，关键控制流交给代码。[OpenAI Agents SDK：Agent orchestration](https://openai.github.io/openai-agents-python/multi_agent/)

### 3.2 上下文窗口不是可靠记忆，而是竞争注意力的工作区

把更多规则、代码和历史全部塞进上下文，不等于模型能同等可靠地使用它们。*Lost in the Middle* 的实验显示，相关信息在长上下文中的位置变化会显著影响模型表现，位于中间的信息更容易利用不足。[Lost in the Middle](https://arxiv.org/abs/2307.03172)

工程上不能只问「是否装得下」，还要问「当前决策真正需要什么」。这直接导出：

- Skill 按需加载，不把全部规则常驻上下文。
- task 显式声明 `context_files`，只给 Agent 当前任务需要的材料。
- 每个 task 使用相对新鲜的子 Agent 上下文，减少历史噪声。
- `tasks.md` 保存跨会话进度，ADR 保存长期 intent，不要求聊天记录承担长期记忆。
- 主会话只编排，不吸收每个实现细节，避免后半程决策质量下降。

因此，上下文工程的目标不是「最大化上下文」，而是提高有效信息密度。

### 3.3 LLM 擅长局部生成，不擅长长周期记账

模型可以很好地完成一个边界明确的局部任务，但跨越多个 task 后，还要持续记住依赖、失败次数、合并状态和质量证据，就把语义推理与流程记账混在了一起。

这直接导出：

- 先把大目标拆成可验证 task。
- DAG 决定顺序，模型不临场猜测依赖。
- `tasks.md` 的状态字段是恢复真相源。
- 控制流内核计算「下一步能做什么」，Agent 只解决当前问题。
- 修复轮次设置上限，避免模型在没有新证据时无限自我迭代。

这不是不信任强模型，而是把模型容量留给只有模型擅长的工作。

### 3.4 生成答案不等于证明答案

LLM 输出在语言和逻辑上可以非常自洽，但代码最终是否正确，取决于编译器、测试、运行环境和真实交互。模型的「我已经修复」只是一个待验证命题。

这直接导出双质量门：

```text
LLM Review：需求、设计、调用链、边界和工程规则
Machine Verify：build、test、lint、基线和浏览器行为
```

- Review 不能用来证明代码能运行。
- 测试通过也不能证明实现符合需求或没有设计遗漏。
- 只有两者都通过，才允许推进合并和交付。

LangGraph 的持久化设计同样把运行状态、失败和恢复位置交给 Checkpoint，而不是要求 Agent 自述执行进度。[LangGraph Persistence](https://docs.langchain.com/oss/python/langgraph/persistence)

### 3.5 同一模型自审容易继承同一组假设

实现者在生成代码时已经建立了一套问题解释。让同一执行主体立即自审，容易继续沿用原来的边界、术语和隐含假设。这里的风险不是模型不够聪明，而是审查缺少认知独立性。

工程推论是：审查独立性按风险购买，而不是所有任务固定支付。

- `lightweight`：单个综合 Reviewer，控制固定成本。
- `standard`：仍用单个综合 Reviewer，但扩大到风险可控的跨模块路径。
- `strict`：实现者、Reviewer、Critic、Judge 分离，用不同职责主动寻找反证。
- 机器 Verify 始终独立于 LLM 判断，提供不同性质的证据。

多 Agent 的价值不是「多投票」，而是制造不同的搜索目标：实现者寻找可行解，Reviewer 寻找缺陷，Critic 质疑 Finding，Judge 基于证据裁决。

### 3.6 Agent 的表现取决于 Harness，不只取决于模型

同一个模型面对不同的工具接口、反馈结构和执行环境，完成任务的能力会不同。SWE-agent 的研究结论是：专门设计的 Agent-Computer Interface 会显著影响软件工程 Agent 的行为与表现。[SWE-agent](https://arxiv.org/abs/2405.15793)

这直接导出：

- Command 只负责稳定入口。
- Skill 提供阶段目标、边界和完成条件。
- Agent 定义提供单一审查职责。
- worktree 提供可安全试错的隔离环境。
- 固定 Review 报告格式同时服务人类阅读和遥测解析。
- 工具输出、测试报告和 commit 成为可检查的 Artifact。

新时代工程师设计的不只是 Prompt，而是模型工作的操作系统界面。

### 3.7 强模型应该获得自主权，而不是更多微管理

能力较强的模型已经能在明确目标下完成代码调研、方案选择、实现和修复。如果仍逐 task 等人确认，会把模型降级成高延迟代码补全工具。

但自主权必须建立在边界内：

```text
人决定目标、重要取舍和风险容忍度
脚本守住状态、依赖和机器门禁
LLM 在批准边界内自主搜索实现路径
```

这直接导出「前重后自主」：人在信息价值最高的需求与设计阶段决策，执行阶段不逐步遥控，收尾只看证据、例外和 intent。

### 3.8 治理成本必须随风险变化

Agent、Reviewer、worktree 和上下文加载都有固定成本。如果所有改动都走最强流程，治理开销会吞掉强模型带来的速度收益。

这直接导出：

- 请求即计划的局部改动走 Fast-Path。
- 需求清楚但实现待设计时直接进入 Quick Design，不重复需求访谈。
- Quick Design 把设计与任务概要合并批准，减少无信息增量的等待。
- Standard 使用单个综合 Reviewer。
- 单 task 已完成 task 级 Review 时，不重复整体 LLM Review。
- Strict 只由安全、权限、数据、并发、性能和公共 API 等高风险信号触发。

减重不是放弃质量，而是删除重复证据：同一种证据只收集一次，不同性质的证据继续保留。

### 3.9 并行能力会把共享状态问题放大

LLM 能同时推进多个独立任务，但多个 Agent 直接操作同一工作区，会把文件覆盖、状态竞争和失败污染放大。并行只有在依赖和写集合明确时才安全。

这直接导出：

- `depends_on` 决定 DAG 波次。
- `lint_task_deps.py` 检查循环、悬空依赖和文件重叠。
- Standard 保留 worktree，因为本框架的真实使用方式包含频繁并行调整。
- 每个 worktree 独立 Review、Verify 和 commit，通过后才合并。
- 下游任务不能在上游尚未合并时提前开始。

worktree 不是 Git 仪式，而是把 Agent 的写操作变成隔离事务。

### 3.10 代码能保存结果，不能完整保存原因

一次决策完成后，代码只能展示「最后选择了什么」，无法可靠还原当时的约束、放弃方案和风险判断。下一个会话若没有这些 intent，只能重新推断，甚至推导出另一套看似合理的解释。

这直接导出：

- Feature Spec 保存一次变更的目标与权衡。
- ADR 保存不可逆或高影响决策。
- Issue 保存已经验证的故障根因。
- 架构快照只作带时效性的导航，不与代码争夺现状真相。
- intent 在上下文最新鲜时沉淀，而不是几周后凭记忆补写。

逐项沉淀检查之所以保留，是因为它检查的是容易永久丢失、又无法从代码恢复的信息。

## 4. 方法论：保留前后两道人/机门，自主化中间执行

```
┌─ 前期（人把关，保留原 workflow-*）─┐   ┌─ 执行（AI 自主并行，新增）─┐   ┌─ 收尾（人把关）─┐
│ requirements-clarification        │   │ 并行 dispatch implementer │   │ 一次性汇总       │
│ system-design / quick-design      │ → │ 每产物过 review           │ → │ intent 沉淀检查   │
│ 任务拆解 + 用户批准 tasks.md       │   │ worktree 隔离 · 不停等     │   │ 交付            │
└───────────────────────────────────┘   └───────────────────────────┘   └─────────────────┘
```

三段的设计原则：

1. **前期设计门按需保留**—— 需求不清走 `requirements-clarification`，需求清楚但实现待设计直接走 `quick-design`，高风险复杂变更再进入完整 `system-design`。Quick Design 将设计与任务概要合并批准，批准后是人把关的最后一道闸。
2. **执行段自主化**（新增的核心）—— 任务清单批准后，**并行**派 implementer subagent 实现，每个产物照样按风险过 `code-review`，但**不再逐 task 停等用户**，全部跑完一次性汇报。质量靠 review 兜底，而非人逐步盯。
3. **收尾人把关** —— 全部完成后一次性汇报 + 按 `project-knowledge` 做交付前 intent 沉淀检查。

> 可选实践：动手前先让一条最核心的端到端链路跑通，验证核心抽象对不对，再批量并行铺开其余功能。属前期设计阶段的事，按需采用，不强制。

## 5. 这套相对 `workflow-*` 改了什么

核心改动仍在**执行段**：把「逐 task 停等的串行执行」换成「下放 Agent 执行（单 Agent / 多 task 并行），批准后自主连跑」，并加入复杂度路由。随后根据真实成本进一步减重：符合「请求即计划」的轻量改动由主会话直改；需求清楚时跳过重复澄清；Quick Design 合并设计与任务批准；Standard 使用综合 Reviewer；单 task 不重复整体 LLM Review。机器 Verify、高风险独立审查、worktree 隔离和 intent 沉淀继续保留。详见 [03-parallel-execution-mode.md](03-parallel-execution-mode.md) 与 [ADR 004](adr/004-lighten-default-workflow.md)。
