# Harness Engineering 实践分析与框架吸收建议

## 结论

两篇文章的共同方法不是「增加更多 Prompt」，而是把 Agent 放进可约束的工程系统：用结构化知识提供上下文，用持久状态承载长流程，用职责隔离降低上下文污染，用确定性程序执行机械步骤，用独立评审和机器门禁阻止错误推进，再把执行证据和失败经验回写成下一轮输入。

`agentic-framework` 已覆盖其中大部分执行骨架：前期设计门、任务 DAG、worktree 隔离、分级 Review、机器验证、有限修复、失败转人工、统一交付证据和 intent 沉淀。当前最值得先补的是 4 个已有缺口：

1. 为执行过程增加轻量、结构化的运行指标。
2. 将高风险任务的评审从实现 Agent 中剥离。
3. 用 Mock 和固定 case 测试工作流控制流。
4. 将知识读取与增量回写挂到明确的工作流节点。

长期不应把框架限制在「大型非生产工具」。更合理的演进方向是「共享内核 + 场景 Profile」：快速开发与生产交付复用同一套需求、设计、任务、执行、Review、Verify 和知识治理内核，再按风险启用不同门禁。完整 P1～P6、部署、灰度、回滚、线上反馈和安全治理应进入生产 Profile 的候选能力，而不是从路线图中排除。

统一 95 分阈值、绑定特定内部平台和固定复制大量 Agent 角色仍不建议直接吸收。这些做法缺少可迁移的量化证据，应由真实生产 case 决定。

## 两篇文章如何构建 Harness Engineering

### 共同骨架

| 能力 | 构建方式 | 工程价值 |
| --- | --- | --- |
| 结构化上下文 | 知识按层组织，按任务渐进加载，不把全库塞进上下文 | 降低噪声和上下文成本 |
| 持久状态 | 阶段、输入、输出、验证结果和失败原因写入共享文件 | 支持暂停、恢复、协作和审计 |
| 确定性管线 | 固定阶段、输入输出契约、完成条件和失败回退 | 降低 Agent 自由发挥造成的漂移 |
| 职责隔离 | 规划、实现、评审、诊断和修复分别处理 | 减少自评偏高和上下文污染 |
| 程序化执行 | JSON 解析、状态转换、worktree、构建、测试和发布交给脚本或强类型程序 | 避免把 Token 消耗在确定性操作上 |
| 质量门禁 | 测试、Review、接口断言、视觉对比和部署检查失败即停止 | 把「完成」从主观判断变成证据判断 |
| 反馈闭环 | 修复后重跑原 case；失败、耗时、Token、成本和知识增量落盘 | 支持回归、复盘和持续改进 |
| 人工边界 | 需求/设计确认、SQL、灰度、回滚和高风险失败转人工 | 防止高影响动作被自动化越权 |

### 原文一：知识底座与确定性编排

原文一将体系拆为「知识库工程 + 端到端开发工程」，核心是让 AI 负责认知，让程序负责确定性执行（原文一第 41～53、397～411 行）。

- **知识底座**：采用总览、业务域、服务三级结构；服务知识覆盖接口、架构、依赖、存储、配置、踩坑和日志等内容（第 83～120 行）。知识由代码事实自动生成，人工补充无法从代码推断的业务知识（第 72～79、123～133 行）。
- **渐进检索**：先用总览缩小范围，再通过索引和 `grep` 定位服务，最后只读取任务需要的文档（第 135～171 行）。
- **状态机**：用 `product-state.json` 管多 Story 的 Fork-Join，用 `e2e-state.json` 管单 Story 的阶段、产物和验证结果；Hook 负责阻止提前结束和恢复会话（第 196～244 行）。
- **职责隔离**：需求、规划、开发、修复、测试、验证、Review、发布和提交由不同角色承担，并限制各自工具与上下文（第 248～283 行）。
- **并行执行**：规划器输出任务 DAG 和预计修改文件，同层任务在独立 worktree 中并行，共享文件、协议或数据库变更串行收口（第 298～333 行）。
- **确定性执行**：JSON、worktree、编译和发布等步骤交给脚本；实践后进一步从 Shell 编排迁向 Go 强类型编排（第 335～350、386～390 行）。这里可迁移的原则是「确定性步骤代码化」，不是「必须使用 Go」。

原文一已有单测、增量覆盖率、代码 Review、接口断言和失败修复闭环，但作者明确承认尚缺稳定性、开发效果和 Token 成本的完整评估体系（第 414～422 行）。

### 原文二：固定管线、纪律层与可观测反馈

原文二把 Harness 定义为模型之外的工作环境，关注执行环境、工具协调、状态、反馈、约束和进展验证，而不是只优化 Prompt（原文二第 13～48 行）。

- **协议层**：规定阶段输入输出、模板、自动校验和增量历史（第 83～91 行）。
- **管线层**：把需求、设计、实现、测试、部署和归档固化为 P1～P6（第 93～103 行）。
- **纪律层**：将 TDD、Debug、Verify、Review 和 Evaluate 做成不可绕过的门禁（第 330～338 行）。
- **研发闭环**：需求 AC 同时服务需求文档和测试用例；设计使用机器可读契约和可定位的改动清单；实现、测试、部署和归档都有完成条件（第 105～256 行）。
- **运行反馈**：`.phase-metrics.jsonl` 记录阶段、耗时、Token、成本和代码变更；独立 Evaluation 评分后才允许推进，失败进入有限轮次整改（第 275～323 行）。
- **知识闭环**：项目级长期知识与单次变更分开，任务结束后只把可复用增量合并回长期知识（第 365～430 行）。
- **上下文控制**：规则常驻、Skill 按需读取、无关任务新开 Session、TODO 文件承载进度（第 469～528 行）。

原文二也明确保留了未解决问题：阶段评分尚未与下游实际效果闭环，知识老化治理未完成，复杂存量项目仍依赖熟悉业务的人初始化，测试本身也可能不可信（第 575～587 行）。文中的统一 95 分、双 95% 和最多 3～5 轮属于团队经验阈值，原文未提供阈值校准数据。

### 建议吸收

#### 架构方向：共享内核 + 场景 Profile

不另起一套完全独立的生产框架。两套框架会复制 Workflow、Skill 和验证规则，长期必然漂移。建议保留共享内核，通过 Profile 提升或降低治理强度。

| 能力 | `rapid` Profile | `production` Profile |
| --- | --- | --- |
| 适用对象 | 原型、内部工具、低风险非生产系统 | 大型生产系统、关键业务变更 |
| 人工门 | 需求、设计和 tasks 批准 | 增加发布、数据变更、灰度和回滚确认 |
| Review | 按风险分级，允许轻量档 | 高风险任务强制独立 Review 与最终集成 Review |
| Verify | build/test/lint、基线和 Spec Drift | 增加安全、兼容、迁移、性能、容量与回滚验证 |
| 执行 | Fast-Path + 自主并行 | 限制 Fast-Path；高风险变更分阶段推进 |
| 状态 | `tasks.md` + 交付证据 | 增加发布状态、环境、审批、制品和回滚点 |
| 可观测性 | 本地轻量指标 | 审计日志、阶段指标、告警与线上反馈闭环 |
| 知识 | ADR、Spec、Issues、经验沉淀 | 增加运行手册、SLO、故障复盘和变更影响记录 |

Profile 必须表达门禁集合和风险策略，不能只是两份重复 Prompt。

#### P0：高风险任务采用独立评审 Agent

**状态**：已吸收。

**问题**：当前 owner 可以「实现 + 自审 + 自修」，同模型家族自审存在共同盲区；框架自己的批判性评估也将其列为待办（[`07-critical-review.md`](07-critical-review.md)，第 23、43 行）。

**最小吸收方式**：

- `review_profile: strict` 时，Review 必须由 owner 之外的独立 Agent 执行。
- owner 只接收 finding 并修复，不参与最终保留/驳回裁决。
- Fast-Path 和轻量档不扩大 Agent 数量。

**成功标准**：严格档执行记录能证明 implementer、reviewer、judge 不是同一执行主体。

#### P0：增加轻量运行指标，而不是建设监控平台

**问题**：框架已有交付证据，但没有统一记录阶段耗时、重试、人工介入和 Token/成本；现有批判性评估已指出缺少预算闸和 spend 留痕（[`07-critical-review.md`](07-critical-review.md)，第 25、44 行）。

**最小吸收方式**：每次标准流程追加一条本地 JSONL 记录：

- feature/task 标识。
- 风险档位与 Agent 数量。
- 开始/结束时间和墙钟耗时。
- Review/Verify 重试次数。
- 最终状态：完成、需人工、阻塞。
- Token/成本：平台能提供时记录；不能提供时写 `unknown`，禁止估算。

先用数据判断是否需要预算上限，不预建 Dashboard、Report API 或集中式平台。

**成功标准**：连续 10 个真实任务可回答「哪一阶段最慢、哪个档位最常返工、人工介入发生在哪里」。

#### P1：用 Mock 和固定 case 测控制流

**问题**：Tier 0/1 已验证结构和触发，但「并行与串行效果」「Workflow 是否在失败时正确回退」尚未实测（[`07-critical-review.md`](07-critical-review.md)，第 27～28、46～47 行）。

**最小吸收方式**：

- 为调度流程提供假 Agent 结果和假 Verify/Review 输出。
- 固定测试 5 类控制流：依赖分波、单任务失败隔离、下游阻塞、修复次数耗尽、会话中断恢复。
- 确定性状态转换用脚本断言；只有输出质量才使用 Rubric。
- 稳定后再选择 1～2 个真实任务做 Tier 2 改前/改后 A/B，不直接进入 Tier 3 自进化。

**成功标准**：无需调用真实模型即可重复验证关键状态转换；同一输入多次运行得到相同流程结果。

#### P1：把知识读写挂点接入 Workflow

**问题**：框架已设计个人知识库和 intent 沉淀，但 `AGENTS.md` 常驻说明只是弱信号；现有方案也明确要求将读写挂到工作流步骤（[`09-personal-knowledge-base-plan.md`](09-personal-knowledge-base-plan.md)，第 255～273 行）。

**最小吸收方式**：

- 需求/设计前读取相关 ADR、历史 Spec、Issues 和共用知识索引。
- Troubleshooting 前读取历史故障。
- 交付门只检查本次是否产生可复用知识，由人确认是否晋升。
- 继续坚持单一权威来源和两跳定位，不复制整份任务文档到长期知识。

**成功标准**：新任务能引用已有结论及来源；任务结束能明确区分「只留任务记录」与「进入长期知识」。

### 生产 Profile 候选能力

| 能力 | 两篇文章提供的启发 | 落地前需要补足 |
| --- | --- | --- |
| 完整 P1～P6 | 从需求、设计、实现、测试走到部署和归档，每阶段有输入输出契约 | 映射到框架现有 Workflow，避免机械复制阶段 |
| 发布与回滚控制 | SQL、灰度、回滚保留人工确认，失败拉取运行日志 | 环境隔离、审批策略、制品追踪、回滚演练 |
| 线上反馈闭环 | 告警或巡检触发采证、诊断、修复、原 case 回归和知识归档 | SLO、告警降噪、故障分级和人工接管边界 |
| 生产级验证 | 接口、UI、日志和部署结果共同决定是否推进 | 安全、性能、容量、兼容性、数据迁移与灾备测试 |
| 全链路审计 | 阶段状态、指标、重试和证据持久化 | 不可篡改审计、身份、权限和凭证治理 |
| 知识生命周期 | 长期知识与单次变更分离，任务结束增量回写 | 新鲜度、淘汰、责任人和故障复盘治理 |

这些能力属于长期路线，但不应一次性全部加入共享内核。应先选一个真实生产变更，逐项验证门禁价值和执行成本。

### 暂不直接吸收

| 做法 | 不吸收原因 | 重新评估条件 |
| --- | --- | --- |
| 所有阶段统一 ≥95 分 | 原文没有阈值校准、误判率和跨项目基线 | 积累稳定 case，并证明分数与下游缺陷相关 |
| 固定复制 12 类 Agent | 角色数量不是目标，会增加 Token 和协调成本 | 真实任务证明职责冲突无法靠现有角色解决 |
| 建集中式指标平台 | 当前没有足够运行数据，JSONL 已能验证价值 | 本地数据达到需要跨项目聚合的规模 |
| 绑定 TAPD、内部发布平台或特定 CLI | 工具专用且外部复现性弱 | 形成平台无关接口和至少两个实现 |
| 全自动知识淘汰或自进化 | 文章自身尚未解决，框架也缺稳定评测集 | 先完成可追溯知识治理和 Tier 2 基线 |

## 推荐落地顺序

1. **先做独立评审边界**：它直接修补高风险任务质量盲区，不引入新平台。
2. **再记录轻量指标**：先获得真实成本和失败数据，再决定预算策略。
3. **补控制流测试**：用 Mock 验证失败、重试、阻塞和恢复，证明 Harness 自身可靠。
4. **接知识读写挂点**：复用已有 `09-personal-knowledge-base-plan.md`，避免另建一套知识体系。
5. **定义 Profile 契约**：明确共享内核、`rapid` 与 `production` 的门禁差异，但先不复制 Skill。
6. **选择一个生产试点**：用真实变更验证发布审批、回滚、生产验证和线上反馈闭环。

明确停止点：前 4 项完成后先运行 10 个真实任务；生产 Profile 先跑 1 个低爆炸半径的真实变更。没有数据证明新缺口前，不继续扩 Agent、扩阶段或建设平台。

## 不确定性

- 两篇文章主要是团队实践总结，不是对照实验；其流程有效性不能直接外推到其他团队。
- 原文一的知识检索优于 RAG、原文二的评分阈值和工具选择都缺少量化对照。
- 当前对照包含工作区未提交修改；框架事实以本次分析日期的实际文件为准，后续提交可能使行号和状态变化。
