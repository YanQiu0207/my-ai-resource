# 项目知识沉淀规范

## 核心哲学：只沉淀代码读不出的东西

本框架对「系统真相」分两层，沉淀策略相反：

| 真相 | 谁是权威 | 怎么对待 |
|------|---------|---------|
| **现状**（系统现在怎么做：模块、接口、数据流） | **代码** | **不维护现状文档**。每次需要现状，重新分析代码提取（这是 `workflow-*` 设计工作流的既定动作）。任何手写的「系统现状」文档都会和代码漂移。 |
| **intent**（当初为什么这么做、放弃了什么、什么约束、什么权衡） | **文档** | **必须沉淀**。代码里根本读不出 intent，不沉淀就永久丢失。载体是 **ADR** + 已归档的 feature spec。 |

> 这是本规范与 OpenSpec 式「中央行为规范库」的根本分歧：我们不维护会和代码争真相的「现状规范库」，只沉淀代码无法还原的 intent。

## 加载时机

- 创建或更新 `docs/design-docs/**/spec.md`、`tasks.md` 时加载
- **每次向用户宣布完成 / 交付前加载**（触发交付前沉淀检查；归档 spec 只是其中一种场景）
- 改动涉及不可逆/高影响的架构决策、重大技术选型、放弃某方案时加载
- 用户询问 ADR、spec 归档或文档应放在哪里时加载

## 目录结构约定

```
docs/
├── adr/                             # 架构决策记录 —— intent 沉淀的主载体
│   └── NNN-<title>.md               # 例：001-choose-message-queue.md
└── design-docs/                     # Feature Spec（按功能创建）
    └── <module>/
        └── <feature>/
            ├── spec.md              # 需求 + 设计 + 为什么（intent 快照）
            └── tasks.md             # 任务拆分与进度
```

> 不设 `architecture/` 现状散文目录：系统现状以代码为准，由设计工作流每次重新调研，不靠手写文档保鲜。

## 各类文档的定位与职责

### Feature Spec（`docs/design-docs/`）

每个功能或较大改动对应一个子目录，生命周期跟随功能：

```
Draft → In Review → Approved → Archived
Quick Draft → Approved → Archived
```

- **Spec**：记录「为什么做、做什么、怎么做」。它的「现状分析」部分会随代码过期（无妨，下次重读代码）；它的**「为什么/目标/备选方案/权衡」是 intent 快照，有持久价值**。
- **Tasks**：记录「分几步做，每步的完成条件是什么」，开发过程中实时更新。
- **归档**：先过[交付前沉淀检查](#交付前沉淀检查强制)，再把 spec.md 头部的 `状态` 字段改为 `Archived`，**文件原地保留，不移动**。
  - 原地保留的好处：避免移动文件导致已有引用失效，且按 `<module>/<feature>` 目录天然可检索。

### 架构决策记录（`docs/adr/`）—— intent 沉淀的主载体

记录**代码读不出、且不可逆或高影响的决策**，一旦写定不修改（只更新状态字段）：

```markdown
# NNN: <决策标题>

**状态**: Accepted / Superseded by NNN

## 背景
<!-- 当时面临什么问题，为什么需要做这个决定 -->

## 决策
<!-- 我们选择了什么方案 -->

## 后果
<!-- 这个决定带来了什么，有什么已知的权衡，放弃了什么 -->
```

**何时写 ADR**：
- 引入/放弃某个存储系统、消息队列、框架或技术方向
- 选择某个有显著权衡的架构模式（且权衡理由代码里看不出来）
- 确立团队编码约定
- 引入一条新的红线约束（性能/兼容/安全），其「为什么」需要被后人理解

## 交付前沉淀检查（强制）

> **每次向用户宣布「完成 / 交付」前**（无论是否归档 spec），**必须**执行本检查，并**逐条给出判定结论**——不得跳过、不得默认略过。归档 spec 只是其中一种触发场景。

### 检查什么

逐条自问，判断本次变更里有没有「代码读不出、值得沉淀」的 intent：

| 信号 | 若命中 → 动作 |
|------|--------------|
| 做了不可逆 / 高影响的架构决策 | 写 / 更新一条 ADR |
| 放弃了某个方案，且放弃理由重要 | 写 ADR，或确认 spec 的「备选方案/权衡」章节已写清 |
| 引入了新的红线约束（性能/兼容/安全） | 写 ADR 或在 spec 中明确记录约束与理由 |
| 只是普通功能/配置/bugfix，代码自解释 | 显式判定「无需沉淀」 |

### 判定规则

1. **现状类信息不沉淀**：模块怎么变、接口长什么样、数据流如何——这些是代码的职责，不写进文档。
2. **必须逐条作答**：对照上面的信号表，每个信号逐行给出「命中 → 已沉淀到 <具体 ADR/spec 章节>」或「未命中」。单句笼统的「判定无需沉淀」不构成有效判定；沉默 = 未完成。
3. **文档债务**：若判定需要沉淀但当前无法完成，在 `tasks.md` 建一个「intent 沉淀」任务，不要跳过。

## 文档与代码的同步原则

| 变更类型 | 需要做的 |
|---------|---------|
| 功能开发完成 / 准备归档 | **执行交付前沉淀检查**；通过后 spec 状态改为 `Archived` |
| 不可逆/高影响决策、放弃方案、新红线约束 | 写 / 更新 ADR（intent 沉淀） |
| 系统现状变了（模块/接口/数据流） | **不写现状文档**——现状以代码为准，下次重新调研 |
| 编码规范变更 | 对应 `std-*` skill 文件 |
| 工程流程变更 | 对应 `workflow-*` skill 文件 |

## 与其他工作流的关系

| 工作流 | 产出 | 在哪里 |
|-------|------|--------|
| `workflow-requirements-clarification` | spec.md（1-3 章：背景/目标/需求） | `docs/design-docs/<module>/<feature>/spec.md` |
| `workflow-system-design` | spec.md（4-8 章：设计/备选/权衡） | 同上 |
| `workflow-quick-design` | spec.md（Quick Draft） | 同上 |
| `workflow-code-generation` | tasks.md + 代码 + **交付前沉淀检查** | `docs/design-docs/<module>/<feature>/tasks.md` |
| `self-refinement` | skill 文件更新 | `skills/` 目录下对应 skill |

ADR 不由设计工作流自动创建，而是在**交付前沉淀检查**判定「值得沉淀」时显式创建；若交付时无法立即完成，在 `tasks.md` 中保留为未完成任务。
