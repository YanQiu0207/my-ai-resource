# 资料库穷举研究

## 目标

基于用户指定的本地资料库回答问题。执行时先高召回搜索，建立可审计的候选资料集合，再逐篇阅读全文，最后输出带文章名、路径和证据的综合结论。

不要只根据少量搜索片段、向量检索结果或 top-k chunk 直接回答。

## 执行边界

本技能可能涉及大量资料，必须在子 agent 中执行主体研究流程：

- 主会话只负责确认输入、发起子 agent、接收并转述最终报告。
- 主会话不要直接枚举、搜索、批量阅读全文或综合大量资料。
- 如果当前平台不支持子 agent，停止并说明「本技能要求子 agent 执行，当前环境不满足」。
- 子 agent 只能使用只读方式研究资料库，除非用户明确要求写文件。

### Claude Code 执行分支

在 Claude Code 中使用本技能时，优先采用两层子 agent 结构：

1. **研究协调 agent**：负责确认范围、扩展检索词、枚举资料、建立候选资料集合、排序筛选和汇总最终报告。
2. **资料阅读 agent**：由研究协调 agent 按文章或小批量文章派发，负责阅读全文并返回单篇阅读笔记。

执行约束：

- 主会话只发起研究协调 agent。
- 研究协调 agent 可继续派发资料阅读 agent，但总阅读集合默认最多 20 篇。
- 每个资料阅读 agent 必须输出文章名、路径、相关性、核心观点和关键证据。
- 研究协调 agent 汇总时必须列出未进入前 20 的候选资料数量和排除原因。

### 非 Claude Code 执行分支

在非 Claude Code 中使用本技能时，主会话应创建一个研究型子 agent，并把以下信息传给子 agent：

- 用户问题。
- 资料库路径或文件列表。
- 本技能的完整流程要求。
- 候选文档默认上限：20 篇。

主会话收到子 agent 结果后，只做格式整理和必要澄清，不重新进行大规模阅读。

## 输入要求

必须同时具备：

- 用户问题。
- 资料库位置，例如目录路径、文件列表或明确的资料范围。

如果缺少任一输入，先向用户询问，不要自行猜测资料范围。

## 核心原则

- 只做研究流程，不做资料清洗、格式转换、向量库构建或知识图谱构建，除非用户另行要求。
- 优先使用只读操作：列目录、搜索、读取文件、汇总。
- 搜索只用于发现候选资料，不能把搜索命中片段当成最终证据。
- 对最终相关或部分相关的资料，必须尽量阅读全文。
- 如果文件太大无法一次读完，按顺序分段阅读，并说明采用了分段阅读。
- 不承诺「绝对不遗漏」；改为报告实际扫描范围、检索词、候选资料、已读资料和不确定性。
- 若发现同义词、英文缩写或相关概念可能影响召回，必须加入检索词集合。

## 执行流程

### 1. 确认范围

明确以下信息：

- 资料库路径或文件列表。
- 是否递归扫描。
- 纳入哪些文件类型。
- 用户的原始问题。

如果用户没有指定文件类型，默认纳入常见文本类文件：

- `.md`
- `.txt`
- `.html`
- `.htm`
- `.json`
- `.jsonl`

### 2. 枚举资料库

列出扫描范围内的文件，并记录：

- 扫描路径。
- 文件总数。
- 纳入的扩展名。
- 跳过的文件及原因。

如果资料数量很大，先给出文件总量和后续分批策略。

### 3. 扩展检索词

从用户问题生成小而高召回的检索词集合，至少考虑：

- 原始关键词。
- 中文同义词。
- 英文术语。
- 常见缩写。
- 相关概念。
- 可能出现在标题里的词。

最终报告中必须展示检索词。

### 4. 高召回搜索

分别搜索：

- 文件名。
- 标题或一级、二级标题。
- 正文全文。

优先召回率，宁可多纳入候选资料，也不要过早排除。

### 5. 建立候选资料集合

对搜索命中的资料做第一轮相关性判断，分类为：

- 相关。
- 部分相关。
- 不确定。
- 低相关或不相关。

不要静默丢弃「不确定」资料；应继续阅读足够上下文，或在最终报告中说明。

**主题校验，不要只看关键词命中。** 一篇资料可能与问题共享大量关键词、术语或品牌名，但实际主题是另一回事。判定相关前，必须确认它讨论的核心对象就是问题所指的对象。例如问「Skill 评测」时，一篇讲「代码评审 Benchmark」的资料可能同时出现「评测」「Benchmark」「同一家公司」，但它测的根本不是 Skill。对这类「关键词高度重叠但主题不同」的资料，归入低相关，并在报告中写明排除理由。

### 6. 筛选阅读集合

如果候选资料较多，默认最多选择 20 篇进入全文阅读。除非用户明确提高上限，不要超过 20 篇。

筛选时优先级如下：

1. 标题、文件名或一级标题直接命中核心主题。
2. 正文多次命中核心检索词或同义词。
3. 覆盖用户问题中的不同子主题，避免 20 篇都集中在同一角度。
4. 出现关键定义、方法论、评测指标、流程、案例或结论。
5. 如果资料带发布时间，优先保留更新且内容更完整的资料。

对未进入全文阅读的候选资料，必须在最终报告中说明数量和排除原因，例如「候选过多，按相关性排序后未进入前 20」。

### 7. 逐篇阅读全文

对「相关」「部分相关」「不确定但可能重要」的资料执行全文阅读。

每篇资料记录一份阅读笔记：

- 文章名或文件名。
- 路径。
- 相关性等级。
- 核心观点。
- 与问题直接相关的事实。
- 关键证据位置，例如标题、章节、行号或附近文本。

每篇笔记必须基于本轮实际读到的正文，不能依赖既有印象、文章标题或摘要概括。同一批资料的笔记详尽度要对等：一篇写多细，由它实际承载的相关信息量决定，不能因为「先读到的几篇已经讲过类似主题」就把后面的资料压缩成一两句话。信息密度高的资料就如实展开，宁可笔记长也不要漏掉关键机制、数据或步骤。

如果资料过大：

- 按自然章节或固定范围分段阅读。
- 保持原文顺序。
- 不要只读取搜索命中附近几行就下结论。

### 8. 综合回答

只基于已阅读资料回答用户问题。

输出时先给结论，再给关键证据。需要区分：

- 资料中明确说了什么。
- 多篇资料之间有什么共识。
- 多篇资料之间有什么差异。
- 哪些内容只是推测或仍不确定。

综合时各来源的信息密度要均衡。一篇资料在总结里占多少篇幅，应由它对问题的贡献决定，而不是由阅读顺序或第一印象决定。不要对先读到的资料写得详细、对同样重要的后读资料草草带过。如果一篇资料含独有的机制、数据或案例，必须在综合中保留这些细节，不能只给一句标签式概括。

## 必须输出的报告结构

```markdown
## 结论

...

## 覆盖范围

- 资料库：
- 是否递归：
- 扫描文件数：
- 候选资料数：
- 全文阅读上限：
- 完整阅读资料数：
- 检索词：

## 相关文章

| 文章名 | 路径 | 相关性 | 关键内容 |
| --- | --- | --- | --- |
| ... | ... | ... | ... |

## 综合总结

...

## 单篇文章摘要

### 《文章名》

- 路径：
- 相关性：
- 核心观点：
- 关键证据：

## 排除或低相关资料

| 文章名 | 原因 |
| --- | --- |
| ... | ... |

## 不确定性

- ...
```

如果排除资料数量过多，可以按原因分组汇总，但必须说明没有逐条列出的数量。

## 质量门禁

最终回答前逐项检查：

- 是否列出扫描范围。
- 是否列出检索词。
- 是否列出候选资料数量。
- 是否逐篇阅读最终相关文章。
- 是否给出文章名和路径。
- 是否区分相关、部分相关、低相关和不确定资料。
- 是否避免了无来源断言。
- 是否明确说明覆盖限制和不确定性。
- 是否对「关键词重叠但主题不同」的资料做了主题校验，而非仅按命中纳入。
- 各篇笔记是否基于实际全文、详尽度对等，没有凭印象压缩。
- 综合总结的信息密度是否均衡，含独有机制 / 数据的资料是否保留了细节，没有厚此薄彼。

## 用户提示词模板

```text
使用 $corpus-research。
问题：……
资料库：……
要求：请在子 agent 中高召回搜索，最多选择 20 篇最相关资料逐篇阅读全文，最后按文章名输出总结和证据。
```
