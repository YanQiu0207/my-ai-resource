# corpus-research 技能的评测方案

> 记录我们为 corpus-research 这个技能搭建并真实验证过的一整套评测方法。先讲采用的方案全貌，再逐个展开关键点（含通俗原理说明与资料出处）。面向不熟悉评测技术细节的读者写。
> 配套实现：`skills/claude-code/corpus-research/evals/`（含 README、GT、check 脚本、报告夹具）。

---

## 一、采用的方案（全貌）

一句话：**把「这个技能跑得好不好」变成可重复、可量化、可回归的数字，并用真实运行的结果反过来改进技能。**

为什么需要它：corpus-research 这类技能的「好」不是对错题。它做一次研究，可以召回得全 / 漏、判主题准 / 混、笔记详 / 略——是一条光谱，不是 boolean。传统软件那套 `assert 输出 == 预期` 的单元测试在这里不够用。最早把这件事讲透的是腾讯《你的 Harness 工作流真的在进步吗》提出的「从测试到考试」：**测试**验证「对不对」（二值判定），**考试**评估「好不好」（多维打分 + 证据 + 改进建议）。

方案是**三层评测 + 一个驱动**：

| 层 | 评什么 | 怎么评 | 能回答的问题 |
| --- | --- | --- | --- |
| A 静态规则 | 技能说明书（SKILL.md）里关键规则还在不在 | 文本匹配，零成本 | 我新加的规则有没有被后来的修改删掉 |
| B 行为召回 | 真跑一次后，该召回的文章漏没漏、不该召回的混没混 | 真执行技能 + 脚本核对 | 会不会遗漏重要知识 |
| C 事实覆盖 | 报告有没有覆盖关键事实点 | 关键词核对 | 该讲的核心点讲到没 |
| 驱动：真跑改进 | — | 真执行 → 评测 → 据结果改技能或改答案集 | 这一版到底有没有变好 |

底座对接了开源框架 **skill-evolver**（腾讯《让 Skill 自己训练自己》一文的开源实现，本地路径 `E:\work\skill-evolution\skill-evolver-fork`），复用它的评测引擎和「LLM 只做二分类、程序算分」的设计哲学。

## 二、关键点展开

### 2.1 核心哲学：LLM 只做二分类，程序算分

**通俗理解**：想象判一场作文比赛。如果让一个评委又判「这段扣题没」又直接给总分，他今天心情好可能多给 5 分——分数不稳。更好的做法：评委只回答一连串「是 / 否」小问题（扣题了吗？有错别字吗？），总分由固定的计分规则（程序）算。同样的「是 / 否」永远算出同样的分。

这是 skill-evolver 的第一原则（出处：`skill-evolver-fork/.../references/eval_strategy.md`，原文「LLMs perform binary classification only; programs compute scores」）。好处是消除大模型不确定性带来的分数漂移。我们的全部 check 脚本都守这条——能用程序判的（文章名在不在、关键词有没有）一律程序判，绝不让大模型给「总分」。

**为什么重要**：skill-evolver 那篇实测，同一技能、同一份答案，让 Claude 评 4 次，分数在 0.79~0.92 之间漂移。解法是多次取均值，但成本翻倍。我们干脆把能确定性判的全部脚本化，直接绕开这个问题。

### 2.2 A 层：静态规则守护（检查「说明书」本身）

**是什么**：一个技能 = 一份给 AI 看的说明书（SKILL.md）+ 可选的参考资料和脚本。A 层不运行技能，只把说明书全文拼起来，检查里面**该有的规则在不在**。

**通俗类比**：像检查一份菜谱**写没写**「最后放盐」——只看菜谱文字，不下厨。

**怎么做**：用 skill-evolver 的 `LocalEvaluator` 跑一组断言（`contains` / `regex`）。比如我们给 corpus-research 加过三道防护规则（主题校验、对等笔记、密度均衡），就写成断言放进 `regression`（回归守护）组——以后谁把这些规则改没了，立刻报红。

**边界**：A 层只能证明「规则文字还在」，证明不了「跑起来有效」。那是 B 层的事。

### 2.3 B 层：行为召回（真跑一次，看漏没漏）

这一层直击最初的诉求「不能遗漏重要知识」。

**怎么做**：
1. 先准备一份「标准答案」（GT，`behavior_gt.json`）：对一个问题，列出**必须召回**的文章（must_recall）和**不该召回**的易混文章（should_not_include）。
2. 真跑一次 corpus-research（真派子 agent 去搜、读、写报告）。
3. 用脚本 `check_recall.py` 核对报告：必召回的漏了几篇？不该召回的混进来没？

**关键技术·锚词**：文章标题里有标点、空格，直接比对容易出错。我们给每篇文章配几个「独特词」当指纹（比如 `SkillTrustBench`、`企业购`、`撕掉了遮羞布`），报告里出现任一指纹就算召回。匹配前先归一化（去空格、忽略大小写），所以跨标点 / 空格也能命中。

**关键技术·相关区**：只在报告的「相关文章 / 单篇摘要」区找指纹，不看「综合 / 不确定性」等叙述区——否则报告里随口提一句「我漏了 X」也会被误判成召回了 X。（这个坑我们真踩过，见 2.8。）

**误纳**：should_not_include 放的是「关键词重叠但主题不同」的易混文章——这是技能最容易出错的地方。比如问「如何评测 Agent」时，SkillTrustBench（Skill 安全评测）、HeartBench（模型评测）都含大量「评测 / Benchmark」关键词，但评的对象不是 Agent。它们若出现在相关区就是误纳。

**成本**：B 层第 1 步「真跑」必须真执行（我们跑一次用了 16.9 万 token、约 8 分钟），框架不替你跑；脚本只负责核对。这是行为评测的固有代价。

### 2.4 C 层：事实覆盖（核心点讲到没）

**是什么**：报告召回了对的文章，不代表把关键事实讲清了。C 层检查报告有没有覆盖关键事实点（比如「5 维 AND 门控不是加权求和」）。

**怎么做**：给每个关键事实配锚词，`check_facts.py` 在报告里查。这是 skill-evolver 所谓 `fact_coverage` 的「确定性近似」——同义改写不算覆盖。要做到语义级判断（改写也算），就得请大模型做二分类，那是进阶选项。

### 2.5 GT 与数据划分：防止「背答案」

**问题**：如果只用一个问题反复调技能，可能把技能改成「只对这个问题好」——像学生背了这套题的答案，换题就废。这叫过拟合。

**解法**：借 skill-evolver 的数据划分思路：
- `dev`：调技能时能看的题。
- `holdout`：**藏起来不给调**的题，最后才用来验，看是真会还是背答案。
- `regression`：守护「老规则别退化」的题。

我们现在有 **10 个 case**，覆盖 Skill 评测、token 优化、Harness Engineering、Agentic Engineering、Multi-Agent、从零搭建 Agent、Agent 评测六大主题。其中 token、Agentic、搭建 Agent 是跨主题 holdout——故意用和「Skill 评测」八竿子打不着的主题，最能验出过拟合。

**铁律**：must_recall 必须是**真读过的文章**才能标，不能凭标题猜。标错答案，评测就会去惩罚正确的技能。

### 2.6 三角色分离：让「考生」看不到答案

**原理**：出题人、考生、判卷人必须分开。同一个 AI 既答题又给自己打分，必然往高了打。这是多篇文章的共识——腾讯《你的 Harness 工作流》（判卷是独立进程、考生看不到 rubric）、阿里 Rubric 那篇（让制作者远离检查者）、《AI Agent & Skill 测评》（人工专门用于校准 LLM 评委）。

**我们怎么做**：真跑 B 层时，让执行的子 agent **只读 SKILL.md、不给它 GT**，它自己搜索判断产出报告；再用 GT 去评判它。这样评测结果真实反映技能质量，而不是「自己出题、自己答、自己评」。

### 2.7 真跑驱动改进：评测反过来修正答案集

**最重要的一条**。skill-evolver 那篇有句话：「GT 质量决定天花板——一个 case 五轮都修不好，先怀疑你的数据，不是技能。」

我们亲历了这一点：真跑 agent-eval 时，子 agent 用全语料搜索，召回了一篇我们 GT 里**漏标**、却被它判为最系统的核心文章《AI Agent & Skill 测评方案及落地实践》。我们扩答案集时只看了 4 篇候选，漏了它。**真跑不只评技能，还反过来补全了答案集。** 这篇我们已补进 GT。

**机制**：skill-evolver 的完整自进化（8 阶段 Loop + 5 维 AND 门控 + trace 诊断）就是把这个「真跑 → 评测 → 改进」自动化循环。我们目前手动走这个闭环；要全自动可接 skill-evolver。

### 2.8 我们实际踩的坑（体现「可归因」）

评测体系的价值之一是「失败能归因」——每次报错都能定位到是技能问题、答案集问题，还是脚本问题。我们这一路抓出的坑，全是后两者（技能本身表现良好）：

1. **否定语境误判**：A 层一条断言「不应包含『绝对不遗漏』」，结果 SKILL.md 里「不承诺『绝对不遗漏』」这句正向表述被误判违规。→ 是断言设计错，改成正向匹配。
2. **锚词在叙述区误命中**：B 层早期把「相关区」当成全文，报告里「我漏了 SkillTrustBench」这句叙述里的指纹被算成召回。→ 收窄到结构化的相关区。
3. **GT 漏标核心文章**：见 2.7。→ 真跑补全。

三个坑全是 GT / 脚本问题，技能本身的主题校验、召回、报告结构在真跑中都达标。这正是「可归因」的体现（出处：腾讯《你的 Harness 工作流》强调的三维归因 `[workflow]` / `[eval]` / `[capability]`）。
