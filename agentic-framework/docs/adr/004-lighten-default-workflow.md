# 004：降低默认工作流重量

**状态**：Accepted

## 背景

框架已经具备 Fast-Path、分级 Review、task 级质量门和最终集成质量门。对 GPT、Claude 等较强模型执行的个人项目，普通任务仍可能承担重复 Review、多 Reviewer 派发和重复批准成本。

本次只减少没有信息增量的重复环节，不删除机器验证、高风险独立审查、DAG 依赖校验和失败接管能力。

## 决策

1. 只有 1 个 task 且合并后没有新增代码改动时，跳过重复的整体 Review；task 级 Review 保留，整体机器 Verify 继续执行。
2. `standard` 默认只调用 `comprehensive-reviewer`，一趟覆盖工程规范、需求/设计符合度、正确性与健壮性；`strict` 保留多 Reviewer、Critic 和独立 Judge。
3. Quick Design 在同一次确认中展示设计与任务概要。用户确认后同时视为设计与任务范围获批，落盘的 `tasks.md` 不再重复审批；范围发生实质变化时必须重新确认。
4. 需求行为与验收标准已经清楚时，不因文件或实现方案尚未确定而强制进入需求澄清：满足低风险局部判据时进入 Fast-Path，需要有限设计时进入 Quick Design；命中高风险条件时仍升级完整流程。

## 本轮未采纳

### Standard 默认取消 worktree

本轮不调整。worktree 是否只在并行或高风险任务启用，需要先核对宿主 Workflow 的隔离合同、恢复路径和合并语义，避免减重时破坏失败隔离。

### 知识沉淀未命中时只输出一句话

本轮不调整。逐项判定是 `project-knowledge` 的交付门合同，是否压缩输出需要单独处理可审计性与交互成本的权衡。

## 后果

- 单 task 不再承担 task 级与整体级双重 LLM Review，但机器 Verify 不减少。
- Standard 的固定 Agent 数下降；发现严格档风险时必须升档，不能由综合 Reviewer 独自放行。
- Quick Design 的人工门从「设计确认后隐式预授权 tasks」改为「设计与任务概要显式合并确认」。
- 「需求清楚」与「实现方案清楚」分离，减少不必要的需求访谈。
- Review 档位、Quick Design 和复杂度路由的触发评测需要同步更新。

## 适用条件

- 适用于本框架的默认个人开发流程。
- 安全、权限、数据迁移、并发、分布式、性能关键路径、公共 API、大范围重构和生产关键路径仍使用 `strict` 或完整设计流程。
