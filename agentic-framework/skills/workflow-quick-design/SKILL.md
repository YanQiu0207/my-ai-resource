# 快速设计

## 适用场景 vs 不适用场景

| 适用 | 不适用（请用 `/system-design`） |
|------|--------------------------------|
| 内部工具（管理后台、运维脚本、异常管理服务） | 核心业务系统的新功能 |
| 小型独立服务（< 3 个核心模块） | 涉及多团队协作的接口设计 |
| 开发者即是设计者、需求相对清晰 | 高风险改动（现有系统的架构变更） |
| 需要快速出方案、不想走完整 spec 流程 | 需要严格 sign-off 的正式设计评审 |

## AI 角色

快速设计中，AI **主动承担设计工作**，而非仅扮演引导者：
- 读完代码背景后直接提出架构方案
- 识别关键风险和权衡，主动说明
- 用户的角色是评审和决策，而非从零描述设计

## 工作流程

### Step 1：收集 Brief（1-2 轮）

向用户询问以下信息（一次问完，不分轮）：

```
请简要描述：
1. 这个工具/服务要解决什么问题？
2. 主要用户是谁（运维？开发？内部系统自动触发）？
3. 有什么关键约束？（性能要求、依赖限制、截止时间等）
4. 是全新项目，还是在已有系统上扩展？
```

用户回答后，如有明显信息缺口，追问一轮；信息足够则直接进入 Step 2。

### Step 2：代码调研（新系统可跳过）

如果是在已有系统上扩展，AI **自主**调研相关代码：
- 读取相关模块结构、接口、依赖关系
- 记录影响设计的技术约束

调研完成后，用一段简短摘要（3-5 条 bullet）告知用户，然后进入 Step 3。

**不要**问用户「现有实现是什么样的」，这些 AI 应该自己读代码获取。

### Step 3：AI 提出设计方案

直接给出一个具体的设计方案，包含：

```markdown
**整体方案**
[一句话描述架构思路]

**核心组件**
- [组件 1]：[职责]
- [组件 2]：[职责]

**主要接口/API**（如适用）
- [接口描述]

**数据流**（如复杂）
[数据如何流转]

**关键权衡**
- [选择 A 的原因，以及放弃 B 的原因]
- [已知风险点]

**运维考量**
- 部署方式：[说明]
- 关键指标：[需要监控什么]

**验收标准**
- [怎样判断这个工具/服务可用]
```

然后问用户：「这个方向符合你的预期吗？有什么需要调整的？」

### Step 4：迭代确认（通常 1-2 轮）

根据用户反馈调整方案，直到用户确认：「好的」「可以」「就这样」类似表述。

**如果用户对方案有重大异议**（不是细节调整，而是方向性不同），重新回到 Step 3。

### Step 5：写入 spec.md

用户确认后，立即写入文件。

先加载 `project-knowledge` 确认文档路径与归档约定。

**文件路径**：`docs/design-docs/<module>/<tool-name>/spec.md`（与标准 spec 路径相同）

使用 [reference/quick-spec-template.md](reference/quick-spec-template.md) 作为模板，只填写有内容的字段，无需关注空白章节。

写完后告知用户文件路径，然后直接进入 Step 6。

### Step 6：拆分任务（tasks.md）

**不询问用户**，直接基于 spec.md 生成 `tasks.md`（与 spec.md 同目录）。

读取 [reference/task_planning_guide.md](../workflow-code-generation/reference/task_planning_guide.md) 并严格按其流程执行：

- **全新项目**：跳过「第二步：代码调研」，直接基于 spec 拆解任务
- **已有系统扩展**：Step 2 的代码调研结论可直接复用，无需重复调研

生成完 `tasks.md` 后，**直接进入 Step 7**，不展示、不等待用户审批。

### Step 7：调用 workflow-code-generation

立即加载 `workflow-code-generation`。

此时 spec.md + tasks.md 均已存在，`workflow-code-generation` 将从步骤 4（加载编码规范）直接进入执行段，不会再次要求创建或审批任务。

## 强制规则

1. **Brief 信息不足时不要开始设计**：缺少问题描述或关键约束时，先问清楚
2. **AI 先提方案**：不要反问「你有什么思路」，直接给方案让用户评审
3. **说明权衡**：每个设计决策要说明为什么选这个，放弃了什么
4. **写入用户确认的内容**：spec.md 内容必须是用户已确认的方案，不要把探讨中的内容写进去
5. **不要写空章节**：quick-spec 模板中未用到的章节直接省略
6. **tasks.md 不询问直接拆**：spec.md 写完后自动进入任务拆分，不问用户「是否需要拆任务」
7. **tasks.md 写完直接调用 code-generation**：不展示、不等批准，直接衔接执行段

## 反模式

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 问用户「你想怎么设计？」 | AI 直接提出方案，用户评审 |
| 把 5 个问题分 5 轮问 | 一次问完所有信息 |
| 方案描述过于抽象（「用微服务架构」） | 给出具体组件和接口 |
| 方案确认后不写文件 | 立即写入 spec.md |
| 强迫用户填所有章节 | 只填有内容的章节 |
| 写完 spec.md 后询问是否拆任务 | 自动拆分，展示后等批准 |
| tasks.md 生成后等用户确认或再说「开始开发」 | 直接调用 workflow-code-generation |
