# 003：review 修复循环收敛（NEEDS_CHANGES 触发 + 增量复审 + 轮次上限）

**状态**：Accepted

## 背景

「问题 → 修复 → review → 新问题 → 修复」循环是执行耗时的最大来源，根因有三：

1. 修复后重跑是全量重审：重新构建上下文、全量派 reviewer、Judge 逐条亲自调研、输出完整报告。reviewer 输出有随机性，每轮全量扫描都可能捞出上一轮未报的新 finding（尤其 P2），循环出现长尾。
2. 规则矛盾：review 通过门槛定义「无 keep 的 P0 / P1 即 PASS，P2 不阻塞」，但调用方规则写「有 finding → 修复后重跑」，字面上 P2 也触发循环。
3. Fast-Path 的循环写成「直到通过」，无轮次上限。

## 决策

1. **触发条件收紧**：仅总体结论为 `NEEDS_CHANGES`（存在 keep 的 P0 / P1）触发修复循环；P2 与 follow-up note 不触发，原样记入报告交用户决定。
2. **第二轮起用复审模式（re-review）**：只核验上一轮 keep finding 是否修复 + 修复 diff 是否引入新问题；只派原 finding 所属维度的 reviewer；禁止对修复 diff 之外提新 finding；输出增量报告。全量扫描是首轮的责任。
3. **轮次上限统一**：全部路径（Fast-Path、per-task owner 循环、步骤 6 最终审核）修复-复审最多 2 轮，仍 `NEEDS_CHANGES` → 标「需人工」终止。

## 放弃的方案

### 修复后全量重审

放弃原因：每轮成本与首轮相同，且 reviewer 随机性使新 finding 持续出现，循环不保证收敛。

### 「有 finding 即修复重跑，直到通过」

放弃原因：与「P2 不阻塞」的通过门槛矛盾；P2 长尾把循环无限拉长；无上限循环在最坏情况下不终止。

### 复审轮允许全范围提新 finding

放弃原因：等价于全量重审，复审模式失去收敛意义；漏报责任应归首轮，靠重跑补漏是用循环成本掩盖首轮质量问题。

## 后果

- 首轮 review 的质量责任变重：复审轮不再兜底首轮漏报。
- P2 不再被自动修复，留给用户决定；交付报告必须完整呈现 P2 与 follow-up。
- 循环有确定上界：最坏 1 次首轮 + 2 次增量复审，之后必然终止（PASS 或需人工）。

## 适用条件

- 适用于所有调用 `workflow-code-review` 后进入修复循环的路径：Fast-Path 第 3 步、owner agent 自闭环、`workflow-code-generation` 步骤 6 最终审核。
- 用户显式要求「把 P2 也修掉」时，修复后仍按复审模式核验，不重开全量。
