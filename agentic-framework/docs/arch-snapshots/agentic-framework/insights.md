# Agentic Framework — 设计洞察

## 核心设计原则

### 1. 前重后自主

```
人把关（需求 + 设计 + tasks 批准）→ AI 自主并行执行 → 人收尾（汇总 + 沉淀）
```

**Why**：大型工具的复杂度需要结构（不能 vibe coding），但生产级门禁太慢（逐 task 停等）。「前重后自主」在结构与速度之间找到平衡——人在上下文最完整时做最高价值的判断，AI 在有约束的范围内自主跑。

### 2. 质量门必须内嵌，不依赖人记得

code review、verify 门禁、intent 沉淀检查全部内嵌进工作流节点，不是可选步骤。

**Why**：如果质量门需要人主动触发，它就会在忙的时候被跳过。内嵌的本质是把「纪律」从依赖人记忆变成依赖流程结构。

### 3. 质量按风险分级投入，不平均

三档 review（lightweight / standard / strict），成本随风险上升。

**Why**：低风险任务 2 个 reviewer 足够；高风险任务 5 个 + critic 才够安全。平均投入要么浪费要么不足，分级是帕累托最优。

### 4. 只沉淀代码读不出的东西

三层真相体系：
- 现状（模块/接口/数据流）→ **代码权威**，不写现状文档
- 架构快照 → **受控例外**，按需落地，显式标注时效性
- Intent（为什么/放弃什么/约束/权衡）→ **文档权威**，必须沉淀

**Why**：任何手写的「系统现状」文档都会和代码漂移，维护成本大于收益。唯一无法从代码还原的是 intent——决策背后的上下文、被放弃的方案、当时的约束条件。这些一旦丢失就永久丢失。

### 5. 知识在上下文最新鲜时沉淀

ADR 即时触发（做出不可逆决策时），交付前检查只是补网。

**Why**：上下文冷却后写出来的 ADR 会遗漏细节，质量显著下降。这条原则来自 compound-engineering-plugin 的「问题刚解决时触发」设计的逆向启发。

## 关键取舍与放弃的方案

### 取舍 1：基于 workflow-* 改造，而非迁移到 Superpowers

**选择**：保留 workflow-* 的质量基建，嫁接 Superpowers 的执行形态。

**放弃的方案**：直接迁移到 Superpowers。

**放弃原因**：Superpowers 无分级 review、无 verify 门禁、无知识沉淀机制，迁移会丢掉 workflow-* 比 Superpowers 强的所有部分。workflow-* 唯一的短板是执行段（逐 task 停等），只需改这一处。

### 取舍 2：不维护现状文档（「代码是唯一现状权威」）

**选择**：不写系统架构现状文档；每次需要现状，重新读代码调研。

**放弃的方案**：维护中央规范库（OpenSpec 式「行为规范库」）。

**放弃原因**：现状文档和代码之间存在根本性的漂移问题——代码改了文档不一定更新，文档就成了误导信息源。重新读代码的成本（有 codebase-researcher）比维护漂移文档的成本低。

**适用条件**：这个原则在有高质量代码 + AI 能快速探索代码的前提下成立。如果代码质量差、可读性低，读代码成本会大幅上升，此时现状文档的价值会增加。

### 取舍 3：复杂度路由（Fast-Path vs 标准流程）

**选择**：极轻改动走 Fast-Path（主会话直改），其余走标准流程（agent 下放）。

**放弃的方案**：统一走标准流程。

**放弃原因**：单行改动也起 worktree + agent，开销不值得。Fast-Path 有安全网——发现多文件改动立即退出，路由错误代价低。

### 取舍 4：从 ce-compound 只吸收「触发时机即时化」，不吸收固定章节结构

**选择**：吸收 ce-compound 的「问题刚解决时触发」原则 → 改为 ADR 即时触发。

**放弃**：ce-compound 的 Bug track 固定章节（Symptoms / What Didn't Work / Solution...）。

**放弃原因**：架构快照不是「问题解决记录」，不需要这个形式；ADR 的自由格式更适合承载复杂的架构决策上下文。

## 边界与适用条件

这套框架为以下场景优化：
- 快速构建**大型非生产**工具（功能多、复杂、但容错率高）
- 开发者愿意在前期（需求 + 设计）投入时间
- 使用 Claude Code（或等效 AI）作为执行引擎

以下场景不适合：
- **生产关键系统**：Fast-Path 和自主执行的容错率不够，应回到逐 task 人工确认的原 workflow-*
- **纯探索型原型**：连 spec 都不需要，直接 vibe coding 速度更快
- **团队协作**：tasks.md 作为进度真相源不支持多人并发写回
