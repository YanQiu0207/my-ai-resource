# 个人知识库建设方案

## 结论

个人知识库首版采用「Markdown + Git + 分层索引 + 按需检索 + 增量更新」，暂不引入向量数据库、知识图谱和复杂 Agent 编排。

这套方案优先解决 4 个问题：

- AI 能否快速找到相关知识。
- 答案能否追溯到原始证据。
- 自动提取的事实与人工经验能否分别维护。
- 源内容变化后，旧知识能否被识别和更新。

本文分为两部分：

- 「文章总结」只整理两篇来源文章明确表达的内容。
- 「个人知识库方案」是基于文章原则做的个人场景裁剪，不代表原文已有完整实现。

## 2. 两篇文章总结

### 2.1 《从 AI Coding 到 Harness Engineering 的端到端工程开发实践》

这篇文章把知识库定位为 Harness 的业务上下文底座，为需求拆解、方案设计、编码和验证提供长期知识。其核心方法包括：

1. **自动生成与人工补充并行**
   - AI 从代码中提取服务、接口、依赖和存储等可验证事实。
   - 人工补充业务背景、设计原因、历史包袱和隐性约束。
   - 更新自动生成内容时，应保留人工批注。
   - 证据：第 41～79、123～133 行。

2. **使用三层结构组织知识**
   - 第一层提供全局总览。
   - 第二层提供业务域索引和领域知识。
   - 第三层记录服务级文档。
   - 人工文档和自动文档均参与查询。
   - 证据：第 83～121 行。

3. **渐进式加载而非一次读取全库**
   - 先通过总览识别相关业务域。
   - 再在域索引中缩小范围。
   - 最后只加载问题需要的文档类型。
   - 文章使用结构化目录与 `grep` 支撑该过程。
   - 证据：第 135～171 行。

4. **管理知识新鲜度**
   - 比较文档记录的 Git hash 与源码当前版本。
   - 超出阈值后标记知识过期。
   - 根据代码 diff 增量更新文档，而不是全量重建。
   - 证据：第 174～186 行。

这篇文章更适合指导「项目代码知识库」建设：它解决代码事实如何生成、如何定位、如何判断过期。

### 2.2 《开启 Harness Engineering 探索之旅》

这篇文章把知识库视为研发和线上运营共享的长期记忆，重点是区分长期知识与单次变更，并让任务结果持续进入长期知识。其核心方法包括：

1. **长期知识与单次变更分离**
   - `specs/` 保存项目级长期知识。
   - changes 或 `knowledge-spec/` 保存单次任务产生的知识。
   - 两者通过索引关联，而不是混成一个不断膨胀的目录。
   - 证据：第 380～413 行。

2. **使用 Delta Spec 增量归档**
   - 任务完成后，从实际改动中提取可复用内容。
   - 将增量内容合并回长期 specs，而不是复制完整任务文档。
   - 证据：第 239～256、425～428 行。

3. **建立两级索引和单一事实来源**
   - 查询路径为 `index.md → spec`。
   - 避免没有边界的全局通配搜索。
   - 相同知识只维护一个权威位置，其他位置通过索引引用。
   - 证据：第 391～413、432～443 行。

4. **按生命周期治理知识**
   - 初始化：AI 生成初稿，人工删除过时内容并补充约束。
   - 演进：随着任务持续增量回写。
   - 治理：处理知识老化、淘汰和质量问题。
   - 文章明确说明持续治理和老化淘汰仍未完成。
   - 证据：第 416～430、575～580 行。

这篇文章更适合指导「研发过程知识库」建设：它解决需求、设计、决策、任务结果和故障经验如何形成长期复利。

## 3. 两篇文章的共识与差异

### 3.1 共识

- 知识库不是资料堆积，而是 AI 工作流的长期上下文。
- 结构和索引优先于将全部内容加载进模型。
- 查询应逐层缩小范围，只读取必要内容。
- 自动生成适合处理可验证事实，业务规则和决策原因仍需人工维护。
- 每次任务结束都应检查是否产生了可复用知识，并增量回写。
- 知识与代码或现实不一致会误导 AI，必须支持追溯和更新。

### 3.2 差异

| 维度 | 第一篇 | 第二篇 |
| --- | --- | --- |
| 主要对象 | 服务、接口、依赖、存储等代码事实 | 需求、设计、变更、测试和故障经验 |
| 组织重点 | 全局、业务域、服务三层目录 | 长期 specs 与单次 changes 分离 |
| 更新方式 | Git hash 检查与代码 diff 更新 | Delta Spec 增量合并 |
| 检索重点 | 目录导航与分层 `grep` | `index.md → spec` 两级定位 |
| 尚未解决的问题 | 缺少完整效果评估体系 | 持续治理、老化淘汰和跨项目共享仍在探索 |

## 4. 个人知识库方案

### 4.1 建设原则

1. **原始资料与加工知识分开**：原文用于追溯，加工后的知识用于快速消费。
2. **长期知识与任务过程分开**：任务记录不直接等于长期知识。
3. **事实与判断分开**：可验证事实、个人经验和 AI 推断不能混写。
4. **一个主题只有一个权威位置**：其他文档通过链接引用，避免重复维护。
5. **先建立可用闭环，再升级检索技术**：用真实问题证明文本检索不足后，才引入 embedding 或 RAG。

### 4.2 建议目录

```text
knowledge-base/
    index.md
    domains/
        <topic>/
            index.md
            <knowledge>.md
    issues/
        <issue>.md
    sources/
        articles/
        official-docs/
    changes/
        <date>-<task>/
            requirement.md
            result.md
            delta.md
    projects/
        <project>/
            index.md
    archives/
```

| 目录 | 职责 |
| --- | --- |
| `domains/` | 跨项目复用的方法、概念和实践，本库唯一的长期知识权威层 |
| `issues/` | 已查证的故障与踩坑：现象、根因、修复、适用边界；不收未验证的猜测 |
| `sources/` | 原始文章和官方资料，作为证据来源；现有 `notes/` 迁入此处 |
| `changes/` | 单次任务的输入、实际结果和待归档增量；仅收无项目仓库归属的个人任务，有仓库的项目任务由项目内 `design-docs/` 承担 |
| `projects/` | 每个项目一个 `index.md` 指针：项目知识的权威位置在项目仓库 `docs/`（adr、arch-snapshots、design-docs、issues），此处只放指向和不适合进项目仓库的个人评注 |
| `archives/` | 仅在整个主题或项目退役时整目录移入；单条知识的过期不进这里，见归档规则 |

代码的当前结构、接口和调用链容易随提交变化，不宜长期手工维护；需要保存时，应记录 Git commit，并明确它是可能过期的快照。

#### 位置与接线

- 知识库不放在任何项目内，由用户选定一个独立目录或独立仓库。
- 路径的唯一权威记录在 `~/.claude/CLAUDE.md`，各项目不复制路径。
- 各项目通过 `project-init` 的接线步骤在 AGENTS.md 写入「知识库」章节，声明：路径、检索时机（需求澄清、方案设计、编码实现、排查 bug 前先查根 `index.md`）、两跳定位、禁止全库通配。

#### 归档规则

按内容粒度区分三种情况，不做「单条知识移动归档」：

- 内容过期 → 原地改写，Git diff 即追溯。
- 结论被取代但有历史价值（如「为什么放弃了 X」）→ 标记 `status: deprecated`，原地保留，链接不失效。
- 整个主题或项目不再活跃 → 整目录移入 `archives/`，同时更新根索引，使其退出日常检索范围。

### 4.3 知识条目契约

每篇加工后的知识至少包含：

```markdown
---
status: active
updated: 2026-07-11
source:
source_version:
tags:
---

# 标题

## 结论

## 适用范围

## 关键内容

## 证据

## 限制与不确定性

## 相关知识
```

字段约定：

- `status`：使用 `active`、`stale`、`deprecated` 或 `uncertain`。
- `source`：填写原始文件、代码路径或官方链接。
- `source_version`：填写 Git commit、软件版本或资料获取日期。
- 「证据」：记录文件路径、章节、行号、commit 或官方页面。
- 「限制与不确定性」：记录尚未验证的内容和适用边界。

自动提取区与人工经验区应物理分开，避免自动更新覆盖人工内容，也避免将 AI 推断当成事实。

### 4.4 最小工作流

#### 导入

1. 从一个高频项目或主题开始。
2. AI 根据原始资料生成结构化初稿。
3. 人工删除过时内容，补充隐性约束和个人经验。
4. 为知识添加来源、状态和版本。
5. 更新主题索引和根索引。

#### 查询

```text
根索引
    → 主题或项目索引
        → 相关知识文件
            → 必要证据片段
```

证据不足时再扩大搜索范围，不默认读取整个知识库。

#### 任务归档

每次任务结束检查：

- 是否产生了可跨任务复用的结论？
- 是否出现新的决策、约束或踩坑？
- 实际结果是否推翻了已有知识？
- 哪些内容应合并到长期知识？
- 哪些内容只需保留在任务记录中？
- 对应索引是否已更新？

长期知识按作用域归位：与项目绑定的写入项目仓库 `docs/`（决策进 adr、踩坑进 issues）；剥离项目上下文后仍然成立的晋升到 `domains/`，原位置留引用，保持单一权威。

#### 新鲜度管理

- 项目知识记录 Git commit。
- 外部资料记录版本和获取日期。
- 源发生变化时，先将相关知识标记为 `stale`。
- 根据 diff 更新自动事实，并保留人工经验。
- 无法核实的旧内容标记为 `uncertain`，不继续作为确定事实使用。

### 4.5 被使用与被更新的保证

AGENTS.md 的常驻说明只是弱信号：长会话中的上下文压缩和注意力衰减会让它失效（第二篇第 508～528 行的问题 1）。保证知识库真正被使用和更新，必须把读写挂点显式写进工作流步骤——「把 AI 看不见的东西挪到它一定看得见的地方」（第二篇第 445 行）。

读侧（保证被使用），侵入 agentic-framework 的调研类步骤：

- `workflow-requirements-clarification` Step 1：调研代码前先检索已有 ADR、历史 spec、issues 与共用知识库。
- `workflow-system-design` Step 0 与 `workflow-quick-design` Step 2：设计前先读相关 ADR 和踩坑，已定决策约束设计空间。
- `codebase-researcher` 第二步：调查前先查 arch-snapshots，`vcs_ref` 新鲜的快照作导航起点。
- `troubleshooting` 历史案例：排查前先查项目 `docs/issues/` 与共用知识库 `issues/`。

写侧（保证被更新），挂在既有强制门禁上：

- `workflow-code-generation` 步骤 6 交付前沉淀检查（Fast-Path 与恢复路径均不豁免）承载 ADR、issues 回写与共用库晋升。
- `codebase-researcher` 第五步承载快照落地。
- `troubleshooting` 案例沉淀承载故障回写。
- `workflow-code-generation` 不加读侧挂点：标准流程的知识消费发生在上游设计工作流，Fast-Path 仅限单文件极轻改动。

维护保障：`lint_skill_graph.py` 保证引用目标存在、skill 不成为孤儿节点（L1 结构检查）；「引用存在但工作流步骤没真加载」的语义断链 lint 查不出，靠 `--graph` 输出的引用图做人工或 LLM 审查。

个人库侧没有工作流强制：任务归档清单目前是软约束，是否升级为硬门禁，阶段二依据实际遵从情况评估。

## 5. MVP 实施路线

### 阶段一：跑通文件型知识库

目标：积累 20～50 篇高频知识，验证结构和工作流是否可用。

- 建立根索引和 2～3 个主题目录。
- 只导入真实工作中反复查询的资料。
- 使用 Markdown、Git 和文本搜索。
- 所有加工知识附来源和状态。

完成标准：能够稳定回答「知识在哪里、依据是什么、是否过期」。

### 阶段二：建立增量归档

目标：让实际任务持续产生长期知识。

- 为任务增加 `delta.md`。
- 任务结束时人工确认哪些增量进入长期知识。
- 建立 stale 检查和索引更新清单。

完成标准：新任务能够复用旧任务沉淀的结论，而不依赖聊天历史。

### 阶段三：根据失败案例升级检索

目标：只解决已经出现的检索问题。

- 记录文本检索失败的问题和期望命中文档。
- 区分索引缺失、关键词不一致和语义召回不足。
- 只有语义召回持续失败时，才评估 embedding 或 RAG。

完成标准：技术升级由真实失败案例驱动，并能用同一组问题验证改进效果。

## 6. 验收标准

使用以下 4 类真实问题测试首版知识库：

1. 找到某个工具的安装和验证步骤。
2. 解释某个项目为何采用当前决策。
3. 列出某个主题中已经验证过的踩坑。
4. 判断某条知识是否已经过期。

每个答案应满足：

- 给出知识文件和证据位置。
- 优先使用最新且状态为 `active` 的内容。
- 不把 `uncertain` 内容表述为确定事实。
- 不需要加载大量无关文件。
- 能区分原始事实、人工经验和 AI 推断。

## 7. 首期不做

- 知识图谱。
- 大规模向量数据库。
- 复杂多 Agent 或 DAG 编排。
- 全自动知识淘汰。
- 跨项目统一语义模型。
- 自动将所有聊天记录写入长期知识。

这些能力会增加同步、评估和治理成本，但不是验证个人知识库价值的必要条件。

另有一项显式排除：不对齐 OpenSpec 目录结构。OpenSpec 的核心是维护「当前行为」的中央规范库（`specs/`），与本方案「现状以代码为准」的原则冲突；其 `changes/` 加增量归档的思想已由 `delta.md` 吸收。
